# 07. DB接続設定の外部化

この章では、PostgreSQLの本物のパスワードをJavaソースへ書かず、Windowsの環境変数から読み取る構成を説明します。

---

## 1. なぜ外部化するのか

以前のDAOには、次の定数がありました。

```java
private static final String PASSWORD =
        "YOUR_DB_PASSWORD";
```

GitHub版がプレースホルダーになっている点は安全です。しかし、ローカルで動かすたびにJavaソースへ本物のパスワードを書く運用には、次の危険があります。

```text
本物のパスワードへ書き換える
  ↓
うっかりgit add
  ↓
commit
  ↓
GitHubへ漏えい
```

そこで、コードと秘密情報を分けます。

```text
GitHub上のJavaコード
  └─ 環境変数名だけを知る

Windowsの環境変数
  └─ 本物のDBパスワードを持つ
```

---

## 2. 使用する環境変数

| 環境変数 | 必須 | 未設定時 |
| --- | --- | --- |
| `INVENTORY_DB_URL` | いいえ | `jdbc:postgresql://localhost:5432/postgres` |
| `INVENTORY_DB_USER` | いいえ | `postgres` |
| `INVENTORY_DB_PASSWORD` | はい | DB接続前に`SQLException` |

ローカルの標準構成なら、パスワードだけ設定すれば動きます。

---

## 3. 新しい接続処理

接続作成を`DatabaseConnectionFactory`へまとめます。

```text
ProductDAO ─┐
            ├─ DatabaseConnectionFactory
UserDAO ────┘          ↓
                    PostgreSQL
```

DAOは接続情報を持たず、次を呼ぶだけです。

```java
return DatabaseConnectionFactory.getConnection();
```

---

## 4. 環境変数の読み取り

Javaでは次を使います。

```java
System.getenv("INVENTORY_DB_PASSWORD");
```

パスワードが未設定または空文字なら、分かりやすい例外を投げます。

```java
throw new SQLException(
    "環境変数 INVENTORY_DB_PASSWORD が設定されていません。");
```

パスワードに既定値を設けない理由は、誤って公開用パスワードや空パスワードで動作させないためです。

---

## 5. Windowsで環境変数を設定する

### 推奨：Windowsの画面からユーザー環境変数へ追加

1. Windows検索で「環境変数」と入力する
2. 「システム環境変数の編集」を開く
3. 「環境変数」を押す
4. 上側の「ユーザー環境変数」で「新規」を押す
5. 次を登録する

```text
変数名：
INVENTORY_DB_PASSWORD

変数値：
自分のPostgreSQLパスワード
```

接続先やユーザー名が標準と違う場合だけ、次も追加します。

```text
INVENTORY_DB_URL
INVENTORY_DB_USER
```

---

## 6. Eclipseを再起動する理由

環境変数は、起動済みのEclipseへ自動反映されない場合があります。

```text
Windowsへ環境変数を追加
  ↓
既に起動中のEclipseは古い環境のまま
```

そのため、

```text
環境変数を追加
  ↓
Eclipseを完全終了
  ↓
Eclipseを再起動
  ↓
Tomcatを起動
```

の順にします。

Eclipseから起動するTomcatは、Eclipseが持っている環境を引き継ぎます。

---

## 7. PowerShellで確認する

値そのものを画面へ表示したくない場合は、設定の有無だけ確認します。

```powershell
if ($env:INVENTORY_DB_PASSWORD) {
    "INVENTORY_DB_PASSWORD is set"
} else {
    "INVENTORY_DB_PASSWORD is not set"
}
```

URL：

```powershell
$env:INVENTORY_DB_URL
```

ユーザー名：

```powershell
$env:INVENTORY_DB_USER
```

URLとユーザー名は未設定でも既定値を使います。

---

## 8. 一時的にPowerShellから起動する方法

現在のPowerShellだけへ設定する例です。

```powershell
$env:INVENTORY_DB_PASSWORD = "自分のDBパスワード"
```

そのPowerShellからEclipseを起動すれば、EclipseとTomcatが環境変数を引き継ぎます。

```powershell
& "Eclipseのeclipse.exeへのパス"
```

PowerShellを閉じると、この一時設定は消えます。

> [!CAUTION]
> コマンドへ本物のパスワードを直接書くと、PowerShellの履歴へ残る可能性があります。通常はWindowsの環境変数画面から設定する方が安全です。

---

## 9. 接続時の流れ

```text
DAOがgetConnection()を呼ぶ
  ↓
DatabaseConnectionFactory
  ↓
PostgreSQL JDBC Driverを読み込む
  ↓
環境変数を読む
  ├─ URL未設定 → localhost用既定値
  ├─ USER未設定 → postgres
  └─ PASSWORD未設定 → SQLException
  ↓
DriverManager.getConnection()
  ↓
PostgreSQLへ接続
```

---

## 10. パスワード未設定時

`INVENTORY_DB_PASSWORD`がない状態でDB処理を行うと、次の意味の例外になります。

```text
環境変数 INVENTORY_DB_PASSWORD が設定されていません。
```

以前のように、

```text
password authentication failed
```

だけが出るより、設定漏れだと判断しやすくなります。

---

## 11. Gitへ秘密情報が入っていないか確認する

変更前：

```powershell
git diff
```

プレースホルダーや環境変数名以外に、本物のパスワードがないことを確認します。

例：

```powershell
git grep -n "INVENTORY_DB_PASSWORD"
```

表示されるのは環境変数名であり、値ではありません。

次のようなコードは残してはいけません。

```java
String password = "本物のパスワード";
```

---

## 12. 環境変数も完全な秘密保管庫ではない

環境変数はGitへの誤commitを防ぐには有効です。

ただし、同じ利用者権限で動くプロセスや、PCの管理者から完全に隠せる仕組みではありません。

実務では環境に応じて、次のような仕組みを使います。

- アプリケーションサーバーのDataSource
- Secret Manager
- Vault
- OSの資格情報管理
- コンテナやクラウドのSecret機能

この学習用アプリでは、まず「ソースコードへ秘密を書かない」段階として環境変数を使います。

## 14. 動作確認

### パスワード未設定

```text
INVENTORY_DB_PASSWORDを削除
  ↓
Eclipseを再起動
  ↓
ログインまたは商品一覧
  ↓
設定されていないことを示す例外
```

### パスワード設定済み

```text
INVENTORY_DB_PASSWORDを設定
  ↓
Eclipseを再起動
  ↓
ログイン成功
  ↓
商品一覧・登録・編集・削除が動く
```

### 別のURLを使う場合

```text
INVENTORY_DB_URL
```

へJDBC URLを設定します。

例：

```text
jdbc:postgresql://localhost:5432/postgres
```

---

## 15. この章を読み終えた時点で説明できること

- DBパスワードをJavaソースへ書く危険
- 環境変数名と環境変数値の違い
- `System.getenv()`の役割
- パスワードだけを必須にした理由
- DAOの接続生成を共通化する利点
- Eclipse再起動が必要な理由
- 環境変数がGitへの秘密漏えいを防ぐ仕組み
- 環境変数が完全な秘密保管庫ではないこと
