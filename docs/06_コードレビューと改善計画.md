# 06. コードレビューと改善計画

この章では、`inventory-web-study`の現在のJava・JSP・DAO・Filterを確認し、見つかった内容を次の3種類へ分けます。

```text
A. 直さないと実際の操作で困る不具合
B. 今は動くが、安全性・保守性を上げる改善
C. 学習用として現状のままでよい部分
```

コードレビューで大切なのは、すべてを「バグ」と呼ばないことです。

```text
実際に誤動作する
  → 不具合

現在は正しく動くが、将来危険になりやすい
  → 改善

小規模な学習用として意図的に簡略化している
  → 現状維持でもよい
```

# 第1部　既に直した不具合

## 2. 削除フォームのCSRFトークン不足

### 以前の状態

削除フォームは商品IDだけを送信していました。

```jsp
<input type="hidden"
       name="id"
       value="${product.id}">
```

しかし、`CsrfFilter`はすべてのPOSTで`csrfToken`を確認します。

```text
削除ボタン
  ↓ POST /products/delete
csrfTokenなし
  ↓
CsrfFilter
  ↓
403 Forbidden
```

### 原因

CSRF異常系検証のためにトークンを一時的に外し、その状態が学習用リポジトリへ残っていました。

### 現在の状態

削除フォームへ次が復旧されています。

```jsp
<input type="hidden"
       name="csrfToken"
       value="${sessionScope.csrfToken}">
```

現在のフォームは、

```text
csrfToken
+
商品ID
```

をPOSTします。

### 判定

```text
修正済み
```

これは「改善案」ではなく、通常の削除操作が403になる実際の不具合でした。

# 第2部　優先して直す不具合

## 3. 商品編集で数値変換に失敗すると入力内容が消える

### 現在の処理

`ProductEditServlet#doPost()`では、次を順番に整数へ変換します。

```java
int id = Integer.parseInt(idText);
int price = Integer.parseInt(priceText);
int stock = Integer.parseInt(stockText);
```

その後で`Product`を作ります。

```java
product = new Product(
    id,
    name == null ? "" : name.trim(),
    price,
    stock
);
```

### 問題

`priceText`または`stockText`が整数でない場合、`Product`を作る前に`NumberFormatException`が発生します。

その時点では、

```java
Product product = null;
```

のままです。

catchでは、nullの`product`をJSPへ渡します。

```java
request.setAttribute("product", product);
```

編集JSPは次を読みます。

```jsp
${product.id}
${product.name}
${product.price}
${product.stock}
```

### 起きること

不正な値が送信された場合、

- 商品IDが空になる
- 商品名や入力値が消える
- 編集フォームを正常な状態で再表示できない
- 再送信に必要な商品IDまで失う

可能性があります。

### 分類

```text
実際の入力エラー時に画面が崩れる不具合
```

### 修正方針

数値へ変換する前の文字列をJSPへ戻せるようにします。

例：

```text
idText
name
priceText
stockText
```

をrequest attributeへ保存し、編集JSPでは入力エラー時にその値を優先表示します。

別案として、IDだけ先に厳密に検証し、価格と在庫数の表示用文字列を別属性で保持します。

### 優先度

```text
高
```

## 4. 商品名が100文字を超えると500になる可能性がある

### DB制約

`products.name`は次です。

```sql
VARCHAR(100) NOT NULL
```

### 現在のServlet側チェック

```java
name == null || name.trim().isEmpty()
```

空文字は確認していますが、100文字以内かは確認していません。

### 現在のJSP

商品名inputに`maxlength`がありません。

```jsp
<input type="text"
       name="name"
       ...>
```

### 起きること

101文字以上の商品名を送信すると、Servletのチェックを通過し、DBのINSERTまたはUPDATEで失敗する可能性があります。

```text
利用者の入力ミス
  ↓
本来は入力エラーとして再表示したい
  ↓
SQLException
  ↓
500エラー
```

### 分類

```text
DB制約と画面・Servletの検証が一致していない不具合
```

### 修正方針

Servlet：

```java
if (name.trim().length() > 100) {
    throw new IllegalArgumentException(
        "商品名は100文字以内で入力してください。");
}
```

JSP：

```jsp
maxlength="100"
```

両方へ追加します。

ブラウザ側だけでは改ざんできるため、Servlet側の確認も必要です。

### 優先度

```text
高
```

---

## 5. GitHubから取得した状態ではDBへ接続できない

### 現在の状態

DAOには公開用プレースホルダーがあります。

```java
private static final String PASSWORD =
        "YOUR_DB_PASSWORD";
```

秘密情報をGitHubへ載せない点は正しいです。

しかし、cloneした直後はDBへ接続できません。

### 現在の運用で起きる危険

動かすたびにJavaソースへ本物のパスワードを書くと、

```text
ローカルで一時変更
  ↓
うっかりgit add
  ↓
秘密情報をcommit
```

という事故が起きやすくなります。

### 分類

```text
公開安全性は保たれているが、
実行設定の仕組みが未完成
```

純粋なロジックバグではありませんが、再現性と秘密情報管理の面で最優先の改善です。

### 修正方針

Java 8でも簡単に使える方法として、環境変数から読みます。

例：

```text
INVENTORY_DB_URL
INVENTORY_DB_USER
INVENTORY_DB_PASSWORD
```

Java側では、

```java
System.getenv("INVENTORY_DB_PASSWORD")
```

などで取得します。

さらに、`ProductDAO`と`UserDAO`へ重複している接続処理を、共通クラスへまとめます。

### 優先度

```text
最優先の構成改善
```

# 第3部　安全性を高める改善

## 6. エラーメッセージの出力方法を統一する

### 現在の状態

`login.jsp`では安全側の出力です。

```jsp
<c:out value="${errorMessage}" />
```

一方、商品登録・編集JSPでは直接EL式を出力しています。

```jsp
${errorMessage}
```

### 現時点の危険度

現在の`errorMessage`はServlet側の固定文言です。

```text
価格と在庫数は整数で入力してください。
商品名を入力してください。
```

利用者入力をそのまま含めていないため、現在直ちにXSSになるわけではありません。

### 将来の危険

将来、次のように変更すると危険です。

```java
"不正な商品名です：" + name
```

### 修正方針

すべて次へ統一します。

```jsp
<c:out value="${errorMessage}" />
```

### 分類

```text
現在の脆弱性ではなく、将来事故を防ぐ安全化
```

### 優先度

```text
中
```

## 7. 匿名アクセスでもSessionを作成している

### 現在の状態

`CsrfFilter`はすべてのURLへ適用されます。

```java
@WebFilter("/*")
```

さらに、

```java
httpRequest.getSession(true);
```

を呼びます。

そのため、ログインしていない利用者がトップページやログイン画面をGETしただけでもSessionが作成されます。

### 現在の影響

小規模なローカル学習アプリでは、ほとんど問題になりません。

### 改善案

- POSTフォームを表示する画面でだけトークンを必要にする
- 既存Sessionを先に確認する
- 対象URLを整理する
- CSRFトークン生成と検証の責任を分ける

ただし、Filterの設計が複雑になるため、学習用として急いで直す必要はありません。

### 分類

```text
性能・設計上の改善
```

### 優先度

```text
低
```

---

## 8. パスワード生成ツールのコマンドライン引数

### 現在の状態

```text
PasswordHashGenerator <password>
```

として、パスワードをコマンドライン引数で受け取ります。

### 利点

- ソースへ固定パスワードを書かない
- 使用方法が簡単
- GitHubへ秘密を残しにくい

### 注意点

OSや実行方法によっては、コマンド履歴やプロセス一覧へ引数が見える可能性があります。

### 改善案

```java
System.console().readPassword()
```

で対話入力するか、標準入力から読む方法へ変更します。

ただし、Eclipse Consoleでは`System.console()`がnullになることがあるため、利用環境を考えて実装する必要があります。

### 分類

```text
補助ツールのセキュリティ改善
```

### 優先度

```text
低から中
```

## 9. ログアウト後のブラウザキャッシュ

Sessionを無効化しても、ブラウザの戻るボタンで以前のHTMLが見えることがあります。

サーバー側ではログアウト済みでも、利用者には「まだ画面が見える」ように感じられます。

```text

Pragma: no-cache
```

ただし、再読み込みや新しい操作は`LoginFilter`で拒否されるため、現在も認証状態自体が復活するわけではありません。

### 分類

```text
表示上の安全性・分かりやすさの改善
```

### 優先度

```text
低
```

# 第4部　保守性・構造の改善

## 12. DAOのDB接続処理が重複している

`ProductDAO`と`UserDAO`の両方に、次が存在します。

```text
URL
USER
PASSWORD
Class.forName()
DriverManager.getConnection()
```

### 問題

接続先を変更すると、複数ファイルを直す必要があります。

片方だけ変更すると、

```text
商品DAOは新DB
ユーザーDAOは旧DB
```

のような不一致が起こり得ます。

### 改善案

```text
DbConnectionFactory
または
DbConfig
```

などへまとめます。

```text
ProductDAO ─┐
            ├→ 共通の接続生成
UserDAO ────┘
```

### 優先度

DB設定外部化と同時に対応するのがよいです。

---

## 13. 使用していない`ProductDAO#insert()`

商品登録Servletは現在、

```java
insertWithLog()
```

を使用します。

単純な、

```java
insert()
```

もDAOに残っていますが、現在の画面処理では使われていません。

### 問題

将来、誤って`insert()`を使うと、商品だけ登録されて操作履歴が残りません。

### 選択肢

1. 学習用比較コードとして残し、用途をコメントで明示する
2. 通常処理では使わせないよう削除する
3. `insertWithLog()`だけを公開APIとして残す
4. Service層へトランザクションを移す

### 分類

```text
現在の誤動作ではないが、将来の使い間違い要因
```

### 優先度

```text
中
```

---

## 14. 操作履歴がCREATEだけ

現在は商品登録時のみ、

```text
operation = CREATE
```

を保存します。

商品編集と削除では履歴を保存しません。

### 改善案

```text
CREATE
UPDATE
DELETE
```

をすべて記録します。

ただし、DELETEでは商品を削除した後に`product_id`が`NULL`になる設計です。

商品名や変更前後の値まで監査したい場合は、

```text
target_name
before_value
after_value
```

など、ログ設計そのものを考え直す必要があります。

### 分類

```text
機能拡張
```

---

## 15. Service層

現在：

```text
Servlet
  ↓
DAO
```

小規模な学習用アプリとしては理解しやすい構成です。

機能が増えると、

```text
Servlet
  ↓
Service
  ↓
DAO
```

に分ける選択肢があります。

Serviceへ置くもの：

- 複数DAOをまたぐ処理
- トランザクション境界
- 業務ルール
- 操作履歴
- 権限確認

現段階でService層を入れると、単純な流れが見えにくくなるため、必須ではありません。

---

## 16. 自動テスト

現在はブラウザとpsqlで手動確認しています。

これは処理の仕組みを理解する上では有効ですが、コード変更のたびにすべてを手作業で再確認する必要があります。

追加候補：

```text
PasswordUtilの単体テスト
Productの入力検証テスト
DAOの統合テスト
Servletのテスト
CSRF・ログインの結合テスト
```

最初の1本としては、DBを必要としない`PasswordUtil`のテストが始めやすいです。

---

## 17. エラーログ

現在は`SQLException`を`ServletException`へ包み、Tomcatへ渡しています。

学習用としては原因を追いやすいですが、実運用では次を分けます。

```text
利用者画面：
処理に失敗しました

サーバーログ：
SQLState、例外、スタックトレース、対象処理
```

パスワードやトークンなどの秘密情報をログへ出さないことも重要です。

# 第5部　現状のままでよい部分

このプロジェクトの研修再現という目的に合っています。

```text

+

+
```

## 19. ServletがDAOを直接呼ぶ構成

小規模なCRUD学習用としては、処理の流れが見えやすい構成です。

```text

↓

↓
DAO
↓
```

## 20. PreparedStatement

すべての現在のSQLでは、値を`?`へ設定しています。

```java
VALUES (?, ?, ?)
```

SQLインジェクション対策として、現在の方針を維持します。

## 21. try-with-resources

Connection、PreparedStatement、ResultSetを自動的に閉じています。

```java
try (
    Connection connection = ...;
    PreparedStatement statement = ...;
    ResultSet resultSet = ...
) {
}
```

この方針は維持します。

同じConnectionで、

```text
+

```

を行い、

```text

```

とする現在の実装は、今回の学習目的に合っています。

## 23. ログイン時のSession再作成

古いSessionをinvalidateし、新しいSessionへ`loginUser`を保存しています。

Session固定攻撃対策として、現在の方針を維持します。

## 24. JSPを`WEB-INF`へ置く構成

ブラウザからJSPを直接開かせず、Servletからforwardしています。


MVCの役割分担として維持します。

## 25. CSRF対策

Sessionへランダムトークンを保存し、POSTフォームのhidden値と比較しています。

削除フォームの不足も修正済みです。

# 第6部　改善の実施順


## 28. この章を読み終えた時点で説明できること

- 不具合、改善、学習用の簡略化を分ける理由
- 削除フォームのCSRF不足が実際の不具合だった理由
- 商品編集の数値エラー時に`product`がnullになる問題
- DBの`VARCHAR(100)`とServletの検証を合わせる必要
- 公開用プレースホルダーと実行用DB設定の違い
- エラーメッセージを出力エスケープへ統一する理由
- 匿名GETでもSessionを作る現在の設計
- DAO接続処理の重複
- 未使用の`insert()`が将来の誤用要因になること
- Service層や自動テストが今すぐ必須ではない理由
- 変更を小さいコミットへ分ける理由
