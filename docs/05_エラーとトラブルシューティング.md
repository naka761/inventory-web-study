# 05. エラーとトラブルシューティング

この章は、`inventory-web-study`を作る途中で実際に遭遇したエラーや混乱を、**症状から原因を切り分けるための記録**としてまとめたものです。

Webアプリのエラーは、Javaコードだけが原因とは限りません。

```text
ブラウザ
  ↓
Tomcat
  ↓
Filter
  ↓
Servlet
  ↓
DAO
  ↓
JDBC Driver
  ↓
PostgreSQL
```

さらに開発時には、

```text
Eclipse
Maven
JDK
Git
GitHub
Windowsの認証情報
```

も関係します。

そのため、エラーを見たら最初に、

> どの層まで処理が到達しているか

を考えます。

---

## 1. 最初に確認する5項目

問題が起きたとき、いきなりコードを全部読み直すのではなく、次の順で確認します。

```text
1. ブラウザからTomcatへ到達しているか
2. Tomcatは対象Servletを呼べているか
3. Filterで止められていないか
4. DAOからPostgreSQLへ接続できているか
5. DB処理後のデータをJSPが正しく表示しているか
```

### 1.1 症状と疑う場所

| 症状 | 最初に疑う場所 |
| --- | --- |
| ブラウザが接続できない | Tomcat、ポート、URL |
| 404 | URL、`@WebServlet`、コンテキストパス |
| 403 | CSRF Filter、ログイン状態 |
| 500 | Servlet、DAO、SQL、DB接続 |
| 画面は出るが一覧が空 | SQL、DBデータ、request attribute |
| 日本語だけ化ける | 文字コード設定 |
| Eclipseだけ赤線 | JDK、Maven Dependencies、Tomcat Runtime |
| `mvn`が使えない | CLI Maven、PATH |
| GitHubへpushできない | Git認証、remote URL、権限 |

# 第1部　Tomcat・URL・HTTP

## 2. `localhost:8080`へ接続できない

### 症状

```text
このサイトにアクセスできません
接続が拒否されました
```

### 主な原因

- Tomcatが起動していない
- Tomcatの起動に失敗している
- ポート8080以外で起動している
- 別プロセスが8080を使用している
- URLのポート番号が違う

### 確認

EclipseのServersビューで、Tomcatが`Started`になっているか確認します。

Consoleで、起動失敗やポート競合を確認します。

PowerShellでポート確認：

```powershell
Test-NetConnection localhost -Port 8080
```

### 考え方

```text
8080へ接続できない
  ↓
ServletやDAO以前の問題
```

この段階では、SQLやJSPを修正しても解決しません。

---

## 3. 404 Not Found

### 症状

Tomcatは動いているが、指定URLが見つからない。

### 主な原因

- コンテキストパスが違う
- `@WebServlet`のURLとアクセスURLが違う
- プロジェクトがTomcatへ追加されていない
- Publishされていない
- Servletクラスのコンパイルに失敗している

### URLの分解

```text
http://localhost:8080/inventory-web/products
```

```text
localhost        接続先PC
8080             Tomcatのポート
inventory-web    コンテキストパス
products         ServletのURL
```

`ProductListServlet`の対応：

```java
@WebServlet("/products")
```

### 確認

```text
EclipseのServers
  ↓
Tomcatへinventory-webが追加されているか
  ↓
Publish
  ↓
Tomcat再起動
```

## 4. 403 Forbidden

### このプロジェクトでの主な原因

`CsrfFilter`がPOSTを拒否しています。

```java
if (requestToken == null
        || !sessionToken.equals(requestToken)) {

    httpResponse.sendError(
        HttpServletResponse.SC_FORBIDDEN,
        "不正なリクエストです。");
}
```

### 確認箇所

POSTフォーム内に次があるか確認します。

```jsp
<input type="hidden"
       name="csrfToken"
       value="${sessionScope.csrfToken}">
```

### 実際に起きた例

CSRF検証のため、削除フォームからtokenを一時的に外しました。

期待どおり403になりましたが、検証後に戻し忘れると通常の削除操作まで403になります。

### 切り分け

```text
GETは成功
POSTだけ403
  ↓
CSRFトークンを確認
```

## 5. 400 Bad Request

このプロジェクトでは、商品IDを整数へ変換できないときに返します。

```java
Integer.parseInt(idText);
```

失敗時：

```java
response.sendError(
    HttpServletResponse.SC_BAD_REQUEST,
    "商品IDが不正です。");
```

### 例

```text
/products/edit?id=abc
```

`abc`は整数へ変換できません。

### 原因候補

- URLを手入力してIDを壊した
- hidden inputが欠けている
- フォームの`name="id"`が違う
- 空文字が送信された

## 6. 500 Internal Server Error

### 意味

サーバー内部で例外が発生しました。

このプロジェクトでは、DAOの`SQLException`をServletが`ServletException`へ包んでいます。

```java
catch (SQLException e) {
    throw new ServletException(
        "商品の登録に失敗しました。",
        e);
}
```

### まず見る場所

EclipseのConsoleで、最下部だけではなく、

```text
Caused by:
```

を上へたどります。

### よくある原因

- PostgreSQLが停止している
- DBパスワードが違う
- テーブルが存在しない
- 列名が違う
- NOT NULL違反
- 外部キー違反
- JDBC Driverが読み込めない
- Javaコードの`NullPointerException`

## 11. NOT NULL違反

### 例

```text
null value in column "operation"
violates not-null constraint
```

### このプロジェクトでの意味

rollback検証で、わざと次へ変更したときに発生させました。

```java
logStatement.setString(2, null);
```

これは検証としては成功ですが、通常状態では次へ戻します。

```java
logStatement.setString(2, "CREATE");
```

### 確認

コード内に検証用の`null`が残っていないか確認します。

## 12. 外部キー違反

### 例

```text

```

### 原因

存在しない商品IDやユーザーIDを、操作履歴へ登録しようとしています。

```text

  ↓
app_usersにid=999がない
```

### 確認SQL

```sql
SELECT id, login_id
FROM app_users;
```

```sql
SELECT id, name
FROM products;
```

## 13. Java 8とJava 25の混在

### 症状

- Eclipseでは動くが、コマンドラインでは違う結果
- Mavenだけ失敗する
- Tomcatだけ別のJavaで起動する
- Java 8で使えないAPIを書いてしまう

### 原因

PCには複数JDKがあり、場所ごとに使うJavaが違います。

```text
Windowsの既定Java：Java 25
Eclipseのプロジェクト：Java 8
Tomcat：設定次第
Maven：実行環境次第
```

### 確認箇所

```text

プロジェクトのJRE System Library
Tomcat RuntimeのJRE
java -version
mvn -version
```

### 重要

`pom.xml`の`source=1.8`、`target=1.8`だけでは、実行に使うJDKまで自動でJava 8になるわけではありません。

## 17. PKIX path building failed

### 症状

```text
PKIX path building failed
unable to find valid certification path
```

### 意味

JavaがHTTPS接続先の証明書を信頼できません。

Mavenが外部リポジトリからjarを取得するときなどに発生します。

### 登場人物

```text
MavenまたはEclipse
  ↓ HTTPS
外部リポジトリ
  ↓ 証明書を提示
Java
  ↓ truststoreで信頼確認
信頼経路を作れない
```

### 原因候補

- 社内プロキシがHTTPS通信の間に入っている
- 会社独自の証明書がJDKへ登録されていない
- EclipseとMavenが別JDKを使っている
- JDKごとにtruststoreが違う
- 接続先の証明書経路を検証できない

### 注意

```text
証明書が古い
```

と即断してはいけません。

確認する順番：

```text
誰がHTTPS通信しているか
  ↓
その処理はどのJDKで動いているか
  ↓
そのJDKのtruststoreは何か
```

会社PCでは、独断で証明書を追加せず、研修担当者や管理者の手順に従います。

# 第4部　文字化け

## 18. Webフォームの日本語がDBへ文字化けして保存された

### 症状

ブラウザで、

```text
トランザクション確認
```

と登録したのに、DBへ次のような文字が保存されました。

```text
ãã©ã³...
```

### 原因

`CsrfFilter`が先に次を呼びました。

```java
getParameter("csrfToken")
```

その時点でフォーム本文全体が解析されます。

Servlet側の、

```java
request.setCharacterEncoding("UTF-8");
```

は後だったため間に合いませんでした。

### 修正

`CsrfFilter`で、最初の`getParameter()`より前に実行します。

```java
httpRequest.setCharacterEncoding("UTF-8");
```

### 覚え方

```text
setCharacterEncoding()
  ↓
最初のgetParameter()
```

でなければなりません。

# 第5部　認証・Session

## 21. 正しいパスワードでもログインできない

### 原因候補

- `password_hash`が壊れている
- 末尾に改行や空白がある
- 保存形式が`反復回数:salt:hash`になっていない
- Base64部分が壊れている
- login_idが違う
- DBへ接続できていない

### 実際に起きた原因

`password_hash`末尾に改行が入りました。

修正：

```sql
UPDATE app_users
SET password_hash =
    BTRIM(password_hash, E' \t\r\n')
WHERE login_id = 'admin';
```

### 確認

保存文字列を目視するだけでなく、文字数や末尾空白も疑います。

## 22. ログイン後すぐログイン画面へ戻る

### 原因候補

- Sessionへ`loginUser`を保存していない
- 保存名が違う
- Filterが別名で探している
- Cookieが保持されていない
- Sessionが直後にinvalidateされている

### 正しい対応

保存：

```java
newSession.setAttribute(
    "loginUser",
    user);
```

確認：

```java
session.getAttribute("loginUser")
```

文字列`"loginUser"`が一致している必要があります。

## 23. ログアウト後に戻るボタンで画面が見える

### 原因

ブラウザキャッシュにより、以前のHTMLが一時表示されることがあります。

### 確認

戻った後に、

```text
再読み込み
商品登録
商品編集
```

など、サーバーへ新しいリクエストを送ります。

Sessionが無効なら、LoginFilterによりログイン画面へ戻されます。

## 27. `.gitignore`があるのに秘密情報が載る

`.gitignore`は、指定したファイルをGit管理対象から外すものです。

しかし、Javaファイル内へ本物のDBパスワードを書けば、Javaファイル自体は管理対象なのでpushされます。

```java

        "本物のパスワード";
```

は危険です。

### さらに重要

一度commitした秘密情報は、後から現在ファイルで削除してもGit履歴に残ります。

そのため、**初回commit前に確認する**ことが重要です。

# 第7部　Eclipseプロジェクトのコピー・Import

## 28. コピーしたプロジェクトがImport後に表示されない

### 実際の状況

元プロジェクト：

```text
inventory-web
```

コピー先フォルダ：

```text
inventory-web-study
```

しかし、コピーされた`.project`内の名前が元のままでした。

```xml
<name>inventory-web</name>
```

Eclipseから見ると、既に存在する同名プロジェクトをもう一度Importしようとしています。

### 修正

コピー先の`.project`：

```xml
<name>inventory-web-study</name>
```

へ変更します。

その後、

```text
File
  ↓ Import
General
  ↓ Existing Projects into Workspace
```

から読み込みます。

### 考え方

Windowsのフォルダ名と、Eclipse内部のプロジェクト名は別です。

```text
フォルダ名：inventory-web-study
.project内：inventory-web
```

という不一致があり得ます。

---

## 29. `Copy projects into workspace`を選ぶべきか

コピー先フォルダが既にEclipse workspace内にある場合、さらにコピーする必要はありません。

```text
C:\Users\...\eclipse-workspace\inventory-web-study
```

をそのままImportします。

重複コピーすると、どちらを編集しているか分からなくなる可能性があります。

## 39. 問題解決の記録テンプレート

新しいエラーが起きたら、次の形式で残します。

```text
## エラー名・症状

### 起きた操作
何をしていたか

### 表示されたエラー
正確なメッセージ

### 原因
誰が何をしようとして失敗したか

### 確認したこと
コマンド、設定、DB内容

### 修正
変更したファイル・設定

### 結果
直ったか

### 再発防止
次回どこを見るか
```

「直った手順」だけでなく、**なぜ失敗したか**を残すと、別のエラーにも応用できます。
