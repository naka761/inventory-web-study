# 04. トランザクションとセキュリティ検証

この章は、`inventory-web-study`へ実装した機能について、**コードを見ただけで「安全そう」と判断せず、わざと失敗させたり、攻撃用文字列を入力したりして確認した記録**です。

このプロジェクトでは、次を実際に検証しました。

- 商品登録と操作履歴が両方成功したときの`commit`
- 操作履歴だけ失敗させたときの`rollback`
- PreparedStatementによるSQLインジェクション対策
- `c:out`や`fn:escapeXml`によるXSS対策
- CSRFトークンを外したときの403
- LoginFilterによる未ログインアクセスの遮断
- PBKDF2によるパスワード照合
- 保存ハッシュへの改行混入によるログイン失敗
- ログイン時の古いSession破棄
- ログアウト後に商品画面へ戻れないこと

検証で重要なのは、次の3つを分けることです。

```text
実装したつもり
  ↓
実際に正常系を試す
  ↓
わざと壊して異常系も試す
```

正常系だけ成功しても、失敗時にどうなるかは分かりません。

---

## 1. 検証の基本姿勢

### 1.1 「動いた」だけでは足りない

商品登録が成功しただけでは、次はまだ分かりません。

- 履歴登録が失敗したとき、商品だけ残らないか
- 攻撃用文字列を商品名へ入れたら実行されないか
- CSRFトークンなしでも削除できてしまわないか
- 未ログインでURLを直接開けないか
- パスワードをDBへ平文保存していないか

そこで、対策ごとに**壊れる条件を作って確認**します。

### 1.2 検証結果に残すもの

各検証では、次を記録します。

| 項目 | 内容 |
| --- | --- |
| 目的 | 何を確認したいか |
| 操作 | 何を入力・変更したか |
| 期待結果 | 正しい実装ならどうなるか |
| 実際の結果 | 実際に何が起きたか |
| 判定 | 対策が機能したか |
| 復旧 | 一時変更を元へ戻したか |

---

# 第1部　トランザクション

## 2. トランザクションで守りたいこと

商品登録では、次の2つを実行します。

```text
① productsへ商品を登録
② product_operation_logsへ操作履歴を登録
```

正しい状態：

```text
商品あり
+
その商品のCREATE履歴あり
```

困る状態：

```text
商品あり
+
履歴なし
```

履歴登録に失敗しても商品だけ残ると、

> 誰が登録したか追跡できない商品

が発生します。

そのため、2処理を1つのトランザクションにします。

---

## 3. 実装されているトランザクション

`ProductDAO#insertWithLog()`では、1つの`Connection`を使います。

```java
try (Connection connection = getConnection()) {
    connection.setAutoCommit(false);

    try {
        // productsへINSERT
        // product_operation_logsへINSERT

        connection.commit();

    } catch (SQLException e) {
        connection.rollback();
        throw e;
    }
}
```

### 3.1 `setAutoCommit(false)`

通常のJDBC接続では、SQLが成功するたびに自動確定されます。

```text
商品INSERT
  ↓ 自動commit
履歴INSERT
  ↓ 失敗
```

これでは商品INSERTを戻せません。

そこで、

```java
connection.setAutoCommit(false);
```

により自動確定を止めます。

### 3.2 同じConnectionが必要

```text
同じConnection
├─ 商品INSERT
└─ 履歴INSERT
```

だから、同じ`commit()`または`rollback()`の対象になります。

別Connectionでは、同じトランザクションになりません。

---

## 4. commitの正常系検証

### 4.1 目的

商品登録と操作履歴登録が両方成功したとき、両テーブルへ保存されることを確認します。

### 4.2 実施内容

ブラウザの商品登録画面から、確認用商品を登録しました。

例：

```text
商品名：トランザクション確認
価格：100
在庫数：1
```

### 4.3 コードの流れ

```text
ProductCreateServlet
  ↓ Productを作成
SessionからloginUserを取得
  ↓
ProductDAO.insertWithLog(product, userId)
  ↓
productsへINSERT
  ↓ 自動発行されたproductIdを取得
product_operation_logsへINSERT
  ↓
commit()
```

### 4.4 DB確認

商品：

```sql
SELECT id, name, price, stock
FROM products
WHERE name = 'トランザクション確認';
```

操作履歴：

```sql
SELECT
    id,
    product_id,
    operation,
    operated_by,
    operated_at
FROM product_operation_logs
ORDER BY id DESC;
```

### 4.5 結果

- 商品が`products`へ登録された
- その商品IDを使った履歴が登録された
- `operation`は`CREATE`
- `operated_by`へログインユーザーIDが入った

### 4.6 判定

```text
商品INSERT成功
+
履歴INSERT成功
+
commit成功
```

正常系では、2処理がまとめて確定されました。

---

## 5. rollbackの異常系検証

### 5.1 目的

商品INSERT後に履歴INSERTだけ失敗させ、商品INSERTも取り消されることを確認します。

### 5.2 わざと壊した箇所

通常：

```java
logStatement.setString(2, "CREATE");
```

検証時だけ一時的に変更：

```java
logStatement.setString(2, null);
```

`product_operation_logs.operation`は`NOT NULL`なので、履歴INSERTが失敗します。

### 5.3 期待される流れ

```text
productsへのINSERT
  ↓ 成功
product_operation_logsへのINSERT
  ↓ NOT NULL違反
SQLException
  ↓
rollback()
  ↓
productsへのINSERTも取消
```

### 5.4 確認用の商品

例：

```text
商品名：ロールバック確認
```

### 5.5 DB確認

```sql
SELECT id, name, price, stock
FROM products
WHERE name = 'ロールバック確認';
```

```sql
SELECT
    id,
    product_id,
    operation,
    operated_by
FROM product_operation_logs
ORDER BY id DESC;
```

### 5.6 結果

- 商品登録処理はエラーになった
- `products`に「ロールバック確認」は残らなかった
- `product_operation_logs`にも失敗した履歴は残らなかった

### 5.7 判定

```text
最初のINSERTまで取り消された
  ↓
rollbackが機能した
```

### 5.8 検証後の復旧

検証用の`null`は残してはいけません。

```java
logStatement.setString(2, "CREATE");
```

へ戻します。

現在のコードでは`CREATE`が有効で、`null`の行はコメントとして検証跡が残っています。

---

## 6. rollback時の例外処理

現在のコードは、rollback自体が失敗した場合も考慮しています。

```java
try {
    connection.rollback();

} catch (SQLException rollbackException) {
    e.addSuppressed(rollbackException);
}
```

最初のSQL例外を`e`とし、rollback中の例外を抑制例外として追加します。

```text
最初の原因：履歴INSERT失敗
追加情報：rollbackも失敗
```

最初の原因を消さず、追加の失敗も記録できます。

---

## 7. IDが欠番になってもrollback失敗ではない

rollback後、商品IDが連続しない場合があります。

```text
7
8
10
```

ID 9がないからといって、商品行が残っているとは限りません。

IDENTITYの採番値は、rollbackしても戻らない場合があります。

```text
ID 9を発行
  ↓
INSERT
  ↓
rollback
  ↓
行は消えるが採番値9は再利用しない
```

確認すべきなのは番号の連続性ではなく、対象商品の行が存在するかどうかです。

# 第2部　SQLインジェクション

## 8. SQLインジェクションとは何か

利用者の入力をSQL文字列へ直接連結すると、入力の一部がSQL命令として解釈される可能性があります。

危険な例：

```java
String sql =
    "SELECT * FROM app_users " +
    "WHERE login_id = '" + loginId + "'";
```

攻撃入力の例：

```text
不届きモノの牛さん' OR '1'='1
```

連結後：

```sql
SELECT *
FROM app_users
WHERE login_id =
'不届きモノの牛さん' OR '1'='1'
```

`'1'='1'`は常に真なので、条件全体が真になる可能性があります。

---

## 9. PreparedStatementではなぜ命令にならないのか

現在のコード：

```java
String sql =
    "SELECT id, login_id, display_name, password_hash " +
    "FROM app_users " +
    "WHERE login_id = ?";

PreparedStatement statement =
        connection.prepareStatement(sql);

statement.setString(1, loginId);
```

SQL構造と値を分けています。

```text
SQL構造：
WHERE login_id = ?

値：
不届きモノの牛さん' OR '1'='1
```

JDBC Driverは入力全体を**1個の文字列値**として扱います。

概念的には、

```sql
WHERE login_id =
'不届きモノの牛さん'' OR ''1''=''1'
```

のように、シングルクォートを文字として安全に処理します。

結果は、

> ログインIDが文字どおり  
> `不届きモノの牛さん' OR '1'='1`  
> であるユーザーを探す

という検索になります。

---

## 10. SQLインジェクション検証

### 10.1 目的

攻撃用文字列がSQL命令として実行されず、通常のログインIDとして扱われることを確認します。

### 10.2 入力例

```text
ログインID：
不届きモノの牛さん' OR '1'='1

パスワード：
任意の値
```

### 10.3 期待結果

- 全ユーザーが検索結果にならない
- ログインに成功しない
- SQL構文エラーにならない
- DBテーブルが変更されない

### 10.4 判定

PreparedStatementを使用しているため、入力はSQL構文へ混ざらず、文字列データとして扱われます。

> [!IMPORTANT]
> PreparedStatementを使っていても、SQLの列名や`ORDER BY`部分などを利用者入力で文字列連結すれば別の危険が生じる。  
> `?`で置き換えられるのは値であり、SQL構造を自由入力にしないことも重要です。

# 第3部　XSS

## 11. XSSとは何か

XSSは、利用者が入力したHTMLやJavaScriptを、別の利用者のブラウザで実行させる攻撃です。

危険な商品名：

```html
<script>alert('XSS')</script>
```

JSPがそのままHTMLへ埋め込むと、

```html
<td>
    <script>alert('XSS')</script>
</td>
```

となり、ブラウザがJavaScriptとして実行する可能性があります。

---

## 12. XSS検証

### 12.1 目的

商品名へ`script`タグを登録しても、JavaScriptが実行されず、文字として表示されることを確認します。

### 12.2 登録した商品名

```html
<script>alert('XSS')</script>
```

### 12.3 商品一覧の出力

現在の商品一覧では、商品名を次で表示しています。

```jsp
<c:out value="${product.name}" />
```

`c:out`は、HTMLで特別な意味を持つ文字をエスケープします。

概念上：

```text
<  → &lt;
>  → &gt;
'  → &#39;など
```

ブラウザが受け取る内容は、JavaScriptではなく文字になります。

### 12.4 実際の結果

- アラートは表示されなかった
- 商品一覧に`<script>alert('XSS')</script>`という文字が表示された
- JavaScriptとして実行されなかった

### 12.5 判定

商品一覧の`c:out`による出力エスケープが機能しました。

---

## 13. 入力欄の属性値にもエスケープが必要

登録画面では、入力エラー時の商品名を再表示します。

```jsp
value="${fn:escapeXml(param.name)}"
```

編集画面：

```jsp
value="${fn:escapeXml(product.name)}"
```

HTML属性値の中でも、入力をそのまま出力しないようにしています。

```html
<input value="利用者入力">
```

ここへ引用符などがそのまま入ると、属性を壊して別の属性やタグを挿入される可能性があります。

---

## 14. 現在のJSPで追加確認が必要な箇所

一部のエラーメッセージは次のように直接出力されています。

```jsp
${errorMessage}
```

現在の`errorMessage`はServlet側で固定文言を設定しているため、利用者入力そのものを直接入れてはいません。

ただし、将来、

```java
"不正な商品名です：" + name
```

のように利用者入力をメッセージへ含めると、出力エスケープが必要になります。

安全側へ統一するなら、次のように`c:out`を使います。

```jsp
<c:out value="${errorMessage}" />
```

# 第4部　CSRF

## 15. CSRFとは何か

CSRFは、ログイン中のブラウザを利用し、利用者が意図していないPOSTを別サイトなどから送らせる攻撃です。

重要なのは、ブラウザがCookieを自動送信する点です。

```text
利用者：在庫管理アプリへログイン済み
  ↓
ブラウザ：Session Cookieを持っている
  ↓
悪意あるページが削除POSTを送らせる
  ↓
ブラウザ：Cookieも自動送信
  ↓
サーバーからはログイン中の本人に見える
```

ログイン確認だけでは、利用者本人が意図した操作かどうかまでは分かりません。

---

## 16. CSRFトークンの仕組み

`CsrfFilter`はSessionへランダムなトークンを保存します。

```java
session.setAttribute(
    "csrfToken",
    sessionToken
);
```

JSPはhidden項目として同じ値をフォームへ埋め込みます。

```jsp
<input type="hidden"
       name="csrfToken"
       value="${sessionScope.csrfToken}">
```

POST時、Filterが比較します。

```text
Session内のcsrfToken
          と
フォーム内のcsrfToken
```

一致すれば通過し、不一致または未送信なら403です。

---

## 17. トークン生成

現在のFilterは`SecureRandom`で32バイトを生成します。

```java
byte[] bytes = new byte[32];
RANDOM.nextBytes(bytes);
```

URLで扱いやすいBase64へ変換します。

```java
Base64.getUrlEncoder()
      .withoutPadding()
      .encodeToString(bytes);
```

トークンは推測しにくいランダム値である必要があります。

固定値では意味がありません。

```text
csrfToken=1234
```

のように誰でも予測できる値では、攻撃側も同じ値を送れるためです。

---

## 18. CSRF異常系検証

### 18.1 目的

正規フォームのトークンを削除したPOSTが、Servletへ到達する前に拒否されることを確認します。

### 18.2 実施内容

削除フォームから、検証用に次を一時的に外しました。

```jsp
<input type="hidden"
       name="csrfToken"
       value="${sessionScope.csrfToken}">
```

その状態で削除ボタンを押しました。

### 18.3 期待結果

```text
requestToken = null
  ↓
Sessionのtokenと一致しない
  ↓
403 Forbidden
  ↓
ProductDeleteServletへ進まない
  ↓
商品は削除されない
```

### 18.4 実際の結果

- 403エラーになった
- 削除Servletの処理へ進まなかった
- 商品は削除されなかった

### 18.5 判定

CSRF Filterが機能しました。

### 18.6 検証後の復旧

削除フォームへhiddenトークンを戻す必要があります。

現在のGitHub版`product-list.jsp`では、ログアウトフォームにはトークンがありますが、削除フォームにはトークンが見当たりません。

この状態では、削除操作は検証時と同じく403になります。

修正すべき位置：

```jsp
<form action="${pageContext.request.contextPath}/products/delete"
      method="post">

    <input type="hidden"
           name="csrfToken"
           value="${sessionScope.csrfToken}">

    <input type="hidden"
           name="id"
           value="${product.id}">

    <button type="submit">削除</button>
</form>
```

これは検証用変更の戻し忘れとして、コード修正が必要です。

---

## 19. CSRFトークンとSessionの関係

トークンはSessionへ保存されます。

```text
Session A
└─ csrfToken = ABC

Session B
└─ csrfToken = XYZ
```

別の利用者や別Sessionでは値が違います。

正規フォームはサーバーが生成するため、現在のSessionに対応する値を埋め込めます。

外部サイトは通常、この値を知ることができません。

# 第5部　ログイン制御とSession

## 20. 未ログインアクセスの検証

### 20.1 目的

ログイン画面を経由せず、商品URLを直接入力しても表示できないことを確認します。

### 20.2 操作

ログアウト状態で、ブラウザのアドレス欄へ直接入力します。

```text
http://localhost:8080/inventory-web/products
```

または、

```text
http://localhost:8080/inventory-web/products/create
```

### 20.3 LoginFilterの判定

```java
HttpSession session =
        httpRequest.getSession(false);

boolean loggedIn =
        session != null
        && session.getAttribute("loginUser") != null;
```

未ログインなら、

```java
httpResponse.sendRedirect(
    httpRequest.getContextPath() + "/login");
```

を実行し、`chain.doFilter()`を呼びません。

### 20.4 期待結果

- 商品Servletへ進まない
- ログイン画面へredirectされる
- 商品一覧や登録画面を表示できない

### 20.5 判定

画面のリンクを隠すだけではなく、URL直接入力もFilterで遮断します。

---

## 21. なぜ画面上のボタン非表示だけでは不十分か

HTMLから「商品登録」リンクを消しても、利用者はURLを直接入力できます。

```text
画面にリンクなし
  ↓
/products/createを直接入力
```

そのため、認可・アクセス制御はサーバー側で行います。

```text
JSP：見せ方
Filter：本当に通してよいか
```

---

## 22. ログイン成功時のSession固定対策

認証成功時、古いSessionを破棄します。

```java
HttpSession oldSession =
        request.getSession(false);

if (oldSession != null) {
    oldSession.invalidate();
}
```

その後、新しいSessionを作ります。

```java
HttpSession newSession =
        request.getSession(true);
```

ログインユーザーを保存します。

```java
newSession.setAttribute(
    "loginUser",
    user
);
```

これは、ログイン前から存在したSessionをそのまま認証済みSessionとして使わないためです。

---

## 23. Sessionのタイムアウト

```java
newSession.setMaxInactiveInterval(30 * 60);
```

30分間操作がない場合、Sessionが無効になります。

```text
30分間リクエストなし
  ↓
Session期限切れ
  ↓
loginUserを取得できない
  ↓
LoginFilterが/loginへ戻す
```

これは、席を離れたPCなどでログイン状態が永久に残らないようにする対策です。

---

## 24. ログアウト検証

### 24.1 操作

商品一覧のログアウトボタンを押します。

### 24.2 Servlet

```java
HttpSession session =
        request.getSession(false);

if (session != null) {
    session.invalidate();
}
```

### 24.3 期待結果

- Session全体が無効になる
- `/login`へredirectされる
- その後`/products`へ直接アクセスしてもログイン画面へ戻る

### 24.4 注意

ブラウザの戻るボタンで古いHTMLが一時的に見える場合があります。

それはブラウザキャッシュ表示の可能性があり、サーバー側でSessionが復活したとは限りません。

実際に再読み込みや別の商品操作を行い、Filterで拒否されることを確認します。

# 第6部　パスワード

## 25. 平文パスワード方式の問題

初期の単純な認証では、DBへ次のように保存する形が考えられます。

```text
login_id：admin
password：admin123
```

これはDBを見られた時点で、パスワードがそのまま漏れます。

また、利用者が別サービスでも同じパスワードを使っていれば、被害が広がります。

---

## 26. PBKDF2による保存

現在は`PasswordUtil`でPBKDF2を使います。

設定：

```java
ALGORITHM = "PBKDF2WithHmacSHA256"
ITERATIONS = 120000
SALT_LENGTH = 16
KEY_LENGTH = 256
```

ハッシュ生成：

```text
生パスワード
+
ランダムsalt
+
120000回の反復
  ↓
PBKDF2
  ↓
ハッシュ
```

保存形式：

```text
反復回数:salt:hash
```

saltとhashはBase64文字列です。

---

## 27. saltが必要な理由

同じパスワードでも、利用者ごとにsaltが違えば保存値が変わります。

```text
admin123 + salt A → hash A
admin123 + salt B → hash B
```

saltがなければ、同じパスワードの利用者は同じハッシュになり、同じパスワードを使っていることが分かりやすくなります。

saltは秘密情報ではありません。ハッシュと一緒に保存します。

---

## 28. 反復回数が必要な理由

PBKDF2は同じ計算を多数回繰り返し、1回のパスワード推測に時間をかけさせます。

```text
攻撃者が候補を1個試す
  ↓
120000回相当の計算
```

正規ログインでも計算コストはありますが、総当たり攻撃の速度を下げる狙いがあります。

この反復回数が永遠に最適という意味ではなく、環境や時代に応じて見直す値です。

## 29. パスワード照合の流れ

DBから保存値を取得します。

```java
String passwordHash =
        resultSet.getString("password_hash");
```

照合：

```java
PasswordUtil.verify(
    password,
    passwordHash
);
```

内部処理：

```text
保存文字列を「:」で3分割
  ↓
反復回数を読む
saltをBase64からbyte[]へ戻す
期待するhashをBase64からbyte[]へ戻す
  ↓
入力パスワードから同じ条件でhashを計算
  ↓
MessageDigest.isEqual()で比較
```

## 30. 保存ハッシュの改行混入検証

### 30.1 症状

正しいパスワードを入力してもログインできませんでした。

### 30.2 原因

DBの`password_hash`末尾に改行や空白が混ざっていました。

```text
本来：
120000:salt:hash

実際：
120000:salt:hash\n
```

見た目では気付きにくいですが、文字列としては別です。

### 30.3 修正

```sql
UPDATE app_users
SET password_hash =
    BTRIM(password_hash, E' \t\r\n')
WHERE login_id = 'admin';
```

### 30.4 結果

末尾の不要文字を削除すると、同じパスワードでログインできました。

### 30.5 学び

認証失敗時は、

```text
パスワードが違う
```

だけでなく、

- ハッシュの保存形式
- 区切り文字の数
- Base64として正しいか
- 末尾の空白・改行
- 反復回数
- 使用アルゴリズム

も確認します。

## 31. `MessageDigest.isEqual()`を使う理由

単純な配列比較は、一致しない位置によって処理時間が変わり、わずかな時間差から情報を推測される可能性があります。

```java
MessageDigest.isEqual(
    expectedHash,
    actualHash
);
```

は、セキュリティ用途のバイト列比較に使われます。

## 32. 生パスワードをUserへ入れない

ログイン後の`User`は次だけを持ちます。

```text
id
loginId
displayName
```

生パスワードやハッシュはSessionへ保存しません。

ログイン後に必要なのは、

- 誰がログインしているか
- 画面に何という名前を出すか
- 操作履歴へどのユーザーIDを入れるか

だからです。

## 34. セキュリティ対策は1個で全部を守れない

各対策は、守る対象が違います。

| 対策 | 主に防ぐ問題 |
| --- | --- |
| PreparedStatement | SQLインジェクション |
| `c:out`・`escapeXml` | XSS |
| CSRFトークン | 意図しないPOST |
| LoginFilter | 未ログインアクセス |
| PBKDF2 | DB漏えい時の平文パスワード露出 |
| Session再生成 | Session固定攻撃 |
| Session timeout | 放置されたログイン状態 |
| トランザクション | 複数DB更新の片方だけ成功 |
| DB制約 | 不正・矛盾データの保存 |

PreparedStatementを使ったからXSSも防げる、という関係ではありません。

```text
入力
  ↓ SQLへ入れるとき：PreparedStatement
DB
  ↓ HTMLへ出すとき：出力エスケープ
ブラウザ
```

入口と出口で別の対策が必要です。

## 35. 現在の学習用アプリに残る改善点

### 35.1 削除フォームのCSRFトークン復旧

現在のGitHub版では、削除フォームへhidden tokenを戻す必要があります。

### 35.2 全出力箇所のエスケープ統一

商品名以外も、利用者入力を表示する場合は`c:out`や`fn:escapeXml`を使います。

### 35.3 DB接続情報の外部化

本物のDBパスワードをJavaソースへ書かず、環境変数やローカル設定へ移す必要があります。

### 35.4 認可

現在は「ログイン済みか」だけを確認しています。

管理者と一般ユーザーで権限を分ける場合は、

```text
誰でも一覧閲覧可能
管理者だけ登録・編集・削除可能
```

のような認可処理が必要です。

### 35.5 HTTPS

localhostの学習環境ではHTTPですが、実運用でログイン情報を送る場合はHTTPSが必要です。

### 35.6 エラー画面

例外の詳細を利用者へそのまま見せず、ログへ記録し、画面には一般的なエラーを表示する設計が必要です。

## 37. この章を読み終えた時点で説明できること

- 正常系だけでなく異常系も検証する理由
- 商品と操作履歴を1つのトランザクションにする理由
- `commit()`と`rollback()`をどう実証したか
- rollback後にIDが欠番になっても問題ない理由
- SQL文字列連結が危険な理由
- PreparedStatementが入力を値として扱う仕組み
- XSS用文字列を`c:out`で無害化する考え方
- CSRFがSession Cookieを悪用する攻撃であること
- Sessionとフォームのトークンを比較する理由
- LoginFilterがURL直接入力も遮断すること
- ログイン成功時に古いSessionを破棄する理由
- PBKDF2、salt、反復回数の役割
- 保存ハッシュの末尾改行で認証が失敗した理由
- セキュリティ対策ごとに守る対象が違うこと
