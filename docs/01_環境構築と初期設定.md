# 01. 環境構築と初期設定

このドキュメントは、`inventory-web-study` を数か月後に読み返したときに、環境構築で何を設定したのかだけでなく、なぜその設定が必要だったのか、何を防ぐための設定なのか、どのソフト同士をつないでいるのかを思い出すための教科書です。

このプロジェクトは、Java 8、Servlet、JSP、JDBC、PostgreSQL を使った在庫管理 Web アプリケーションです。ブラウザから Tomcat にアクセスし、Tomcat が Servlet / JSP を動かし、Java の DAO が PostgreSQL に接続します。

## このプロジェクトで実際に使う構成

| 種類 | このプロジェクトの設定 | 補足 |
| --- | --- | --- |
| OS | Windows | 詳細なバージョンは環境によって異なる |
| IDE | Eclipse | Java Web アプリ開発に使用 |
| Java | Java 8 | Temurin JDK 8 を使う前提 |
| Web サーバー / Servlet コンテナ | Apache Tomcat 9 | `javax.servlet` 系のアプリを動かすため |
| データベース | PostgreSQL 18 | Java アプリとは別プロセスの DB サーバー |
| ビルド管理 | Maven | `pom.xml` で依存関係とビルド設定を管理 |
| Servlet API | Java Servlet API 4.0 | `javax.servlet-api:4.0.1` |
| View | JSP / JSTL | JSP で HTML を作り、JSTL で表示ロジックを書く |
| DB 接続 | PostgreSQL JDBC Driver | `org.postgresql:postgresql:42.7.11` |
| 成果物形式 | WAR | Tomcat に Web アプリとして配置する形式 |

この表は、実際の `pom.xml` とこのプロジェクトのコードをもとにしています。Tomcat のインストール場所、Eclipse の設定画面の表示、Windows のサービス名などは環境によって異なります。

## 全体の接続関係

このプロジェクトでは、ブラウザ、Tomcat、Java アプリ、PostgreSQL が別々の役割を持ちます。

```text
ブラウザ
  |
  | HTTPリクエスト
  v
localhost:8080
  |
  v
Apache Tomcat 9
  |
  | Servlet APIでJava Webアプリを呼び出す
  v
Filter / Servlet / JSP
  |
  | JDBC API + PostgreSQL JDBC Driver
  v
localhost:5432
  |
  v
PostgreSQL 18
```

重要なのは、Tomcat と PostgreSQL が同じものではないことです。Tomcat は Web アプリを動かすサーバー、PostgreSQL はデータを保存する DB サーバーです。

## 1. JDKとJREとJavaの関係

### 結論

このプロジェクトの開発には JDK が必要です。JRE だけでは、Java プログラムを実行できても、開発やコンパイルに必要なものが足りません。

### 仕組み

| 用語 | 役割 | このプロジェクトでの意味 |
| --- | --- | --- |
| Java | 言語や実行環境を含む技術全体 | Servlet や DAO を書くための言語 |
| JDK | Java Development Kit | `.java` を `.class` にコンパイルする開発キット |
| JRE | Java Runtime Environment | コンパイル済み Java プログラムを実行する環境 |

Java のコードは、そのまま実行されるわけではありません。

```text
Product.java
  |
  | JDK内のjavacでコンパイル
  v
Product.class
  |
  | Java実行環境で動く
  v
アプリケーションの処理として実行される
```

### なぜ必要か

Eclipse や Maven は、Java のソースコードをコンパイルするために JDK を使います。Tomcat も Java で作られたソフトなので、起動には Java 実行環境が必要です。

### 間違えると何が起きるか

| 間違い | 起きること |
| --- | --- |
| JDK ではなく JRE だけを使う | コンパイルに必要な機能が足りないことがある |
| Eclipse が JDK を見つけられない | Java 標準クラスまでエラーになることがある |
| 想定外の Java バージョンを使う | ビルド結果や実行時の挙動がずれる |

### このプロジェクトで実際に使った設定

このプロジェクトは Java 8 前提です。`pom.xml` でも Java 8 を指定しています。

```xml
<maven.compiler.source>1.8</maven.compiler.source>
<maven.compiler.target>1.8</maven.compiler.target>
```

## 2. PCにはJava 25もあるが、このプロジェクトではJava 8を使う理由

### 結論

PC に Java 25 が入っていても、このプロジェクトでは Java 8 を使います。理由は、プロジェクトのコンパイル対象、Eclipse の設定、Tomcat で動かす前提を Java 8 にそろえるためです。

### 仕組み

Java はバージョンごとに使える文法や標準 API が違います。Java 25 で書けるコードが、Java 8 で動くとは限りません。

```text
Java 25でしか使えない書き方
  |
  v
Java 8対象のプロジェクトではコンパイル不可または実行不可
```

### なぜ必要か

研修や復習用のプロジェクトでは、動作確認する環境を固定した方が原因を追いやすくなります。Java 8、Tomcat 9、`javax.servlet` をセットで考えることで、Java Web 開発の基本構成を安定して学べます。

### 間違えると何が起きるか

| 状態 | 起きる可能性がある問題 |
| --- | --- |
| Eclipse は Java 8、Maven は Java 25 | IDE と Maven で結果が変わる |
| Java 25 の文法を書いてしまう | Java 8 対象でビルドできない |
| Tomcat が想定外の Java で動く | 実行時エラーの原因が分かりにくくなる |

### このプロジェクトで実際に使った設定

`pom.xml` の `source` と `target` は `1.8` です。Eclipse 側も `JRE System Library [JavaSE-1.8]` として扱うのがこのプロジェクトの前提です。

## 3. EclipseのJRE System Library [JavaSE-1.8]の意味

### 結論

`JRE System Library [JavaSE-1.8]` は、Eclipse がこのプロジェクトを Java 8 のプロジェクトとして扱うという意味です。

### 仕組み

Eclipse はこの設定を見て、Java 標準クラスや文法を判断します。

```java
java.lang.String
java.util.List
java.sql.Connection
```

これらは自分で追加したライブラリではなく、Java の標準ライブラリです。

### なぜ必要か

Eclipse が Java 8 のルールでコードをチェックすることで、Java 8 では使えない書き方を混ぜにくくなります。

### 間違えると何が起きるか

| 設定ミス | 起きること |
| --- | --- |
| JavaSE-25 などになっている | Java 8 では使えないコードを書けてしまう |
| JRE System Library が壊れている | `String` や `List` まで解決できないことがある |
| Maven の Java 設定とずれる | Eclipse では通るのに Maven では失敗することがある |

### このプロジェクトで実際に使った設定

このプロジェクトは Java 8 前提なので、Eclipse では `JRE System Library [JavaSE-1.8]` として扱います。

## 4. Tomcatとは何をするソフトか

### 結論

Tomcat は、Java の Web アプリをブラウザから使えるようにするソフトです。HTTP リクエストを受け取り、Servlet や JSP を実行します。

### 仕組み

普通の Java クラスは、ブラウザから直接呼び出せません。Tomcat が間に入り、URL と Servlet をつなぎます。

```text
ブラウザ
  |
  | HTTP
  v
Tomcat
  |
  | URLに対応するServletを呼ぶ
  v
LoginServlet / ProductListServlet / ProductCreateServlet
```

Tomcat は次のようなものを用意します。

| Tomcat が用意するもの | 役割 |
| --- | --- |
| `HttpServletRequest` | ブラウザから届いた情報を Java で扱う |
| `HttpServletResponse` | ブラウザへ返す情報を Java で作る |
| Servlet の実行環境 | `doGet()` や `doPost()` を呼ぶ |
| JSP の実行環境 | JSP から HTML を生成する |
| Session 管理 | ログイン状態などを保持する |

### なぜ必要か

Servlet は、Tomcat のような Servlet コンテナの中で動く前提のプログラムです。Tomcat がなければ、ブラウザからの HTTP リクエストを Java の Servlet へ届けられません。

### 間違えると何が起きるか

| 状態 | 起きること |
| --- | --- |
| Tomcat が起動していない | `localhost:8080` にアクセスしてもアプリが開けない |
| プロジェクトが Tomcat に追加されていない | Tomcat は動いていてもこのアプリは見つからない |
| Tomcat のバージョンが違う | Servlet API の不一致で動かないことがある |

### このプロジェクトで実際に使った設定

このプロジェクトは Tomcat 9 で動かす前提です。理由は、コードが `javax.servlet` を使っているためです。

## 5. Tomcat 9とTomcat 10のjavax.servlet / jakarta.servletの違い

### 結論

Tomcat 9 と Tomcat 10 は、Servlet のパッケージ名が違います。このプロジェクトは `javax.servlet` なので Tomcat 9 を使います。

### 仕組み

| Tomcat | Servlet API | import の例 |
| --- | --- | --- |
| Tomcat 9 | `javax.servlet` | `import javax.servlet.http.HttpServlet;` |
| Tomcat 10 | `jakarta.servlet` | `import jakarta.servlet.http.HttpServlet;` |

Java では、パッケージ名が違うと別のクラスです。

```java
// Tomcat 9系
import javax.servlet.http.HttpServlet;

// Tomcat 10系
import jakarta.servlet.http.HttpServlet;
```

### なぜ必要か

Servlet は Tomcat が呼び出します。アプリ側の Servlet API と Tomcat 側の Servlet API がずれていると、Tomcat はアプリを正しく扱えません。

### 間違えると何が起きるか

| 組み合わせ | 結果 |
| --- | --- |
| `javax.servlet` + Tomcat 9 | このプロジェクトの想定 |
| `javax.servlet` + Tomcat 10 | そのままでは合わない |
| `jakarta.servlet` + Tomcat 10 | Jakarta 系のプロジェクトなら合う |
| `jakarta.servlet` + Tomcat 9 | そのままでは合わない |

### このプロジェクトで実際に使った設定

コードでは `javax.servlet` を import しています。`pom.xml` でも `javax.servlet-api:4.0.1` を使っています。

```xml
<dependency>
    <groupId>javax.servlet</groupId>
    <artifactId>javax.servlet-api</artifactId>
    <version>4.0.1</version>
    <scope>provided</scope>
</dependency>
```

## 6. なぜこのプロジェクトではTomcat 9を使うのか

### 結論

このプロジェクトでは、Tomcat 9、`javax.servlet`、Servlet API 4.0 をセットで使います。

### 仕組み

Tomcat 9 は `javax.servlet` 系の Servlet API を提供します。アプリ側の import と Tomcat 側の API が同じ系統なので、Servlet と Filter を正しく実行できます。

```text
inventory-web-study
  |
  | javax.servlet を使う
  v
Tomcat 9
  |
  | javax.servlet APIを提供
  v
Servlet / Filter が動く
```

### なぜ必要か

このプロジェクトの目的は Java Web 開発の基本を学ぶことです。Tomcat 9 と `javax.servlet` で統一すると、Servlet、JSP、Filter の基本に集中できます。

### 間違えると何が起きるか

Tomcat 10 を選ぶと、`jakarta.servlet` への移行が必要になります。これは別の学習テーマになるため、このプロジェクトでは扱いません。

### このプロジェクトで実際に使った設定

`LoginServlet`、`ProductListServlet`、`CsrfFilter` などは `javax.servlet` を使っています。Tomcat は 9 系を使う前提です。

## 7. localhost、8080、5432の意味と違い

### 結論

`localhost` は自分の PC を指す名前です。`8080` は Tomcat へアクセスするポート、`5432` は PostgreSQL へアクセスするポートです。

### 仕組み

| 値 | 意味 | つながる相手 |
| --- | --- | --- |
| `localhost` | 自分の PC | Tomcat や PostgreSQL が同じ PC で動いている場合に使う |
| `8080` | HTTP 用の待ち受けポート | Tomcat |
| `5432` | PostgreSQL の待ち受けポート | PostgreSQL |

```text
http://localhost:8080/...
  |
  v
Tomcat

jdbc:postgresql://localhost:5432/postgres
  |
  v
PostgreSQL
```

### なぜ必要か

同じ PC の中で複数のサーバーが動くため、ポート番号で接続先を分けます。`localhost` だけでは、Tomcat に行きたいのか PostgreSQL に行きたいのか区別できません。

### 間違えると何が起きるか

| 間違い | 起きること |
| --- | --- |
| ブラウザで `5432` にアクセスする | PostgreSQL は HTTP サーバーではないので画面は開けない |
| JDBC URL に `8080` を書く | PostgreSQL ではなく Tomcat のポートを見に行ってしまう |
| Tomcat のポートが変更されている | `localhost:8080` では開けない |

### このプロジェクトで実際に使った設定

DAO では次の JDBC URL を使っています。

```java
jdbc:postgresql://localhost:5432/postgres
```

Tomcat の HTTP ポートは通常 `8080` を使いますが、Eclipse や Tomcat の設定によって異なる場合があります。

## 8. PostgreSQLが別プロセスのDBサーバーとして動いていること

### 結論

PostgreSQL は Java アプリの中にある部品ではなく、別プロセスとして動く DB サーバーです。

### 仕組み

```text
Tomcat上のJava Webアプリ
  |
  | JDBCで接続
  v
PostgreSQLサーバー
  |
  v
データベース
```

Tomcat を起動しても、PostgreSQL が必ず起動するわけではありません。

### なぜ必要か

商品情報やユーザー情報は、Java のメモリではなく DB に保存します。アプリを再起動してもデータを残すために、PostgreSQL のような DB サーバーを使います。

### 間違えると何が起きるか

| 状態 | 起きること |
| --- | --- |
| Tomcat は起動しているが PostgreSQL が停止 | 画面は開けても DB 処理で失敗する |
| PostgreSQL のポートが違う | JDBC 接続に失敗する |
| DB 名やユーザーが違う | 認証エラーや DB が存在しないエラーになる |

### このプロジェクトで実際に使った設定

DAO では `localhost:5432/postgres` に接続しています。実際の PostgreSQL サービス名や起動方法は Windows の環境によって異なります。

## 9. JDBC APIとPostgreSQL JDBC Driverの役割の違い

### 結論

JDBC API は Java 側の共通ルール、PostgreSQL JDBC Driver は PostgreSQL と実際に話すための実装です。

### 仕組み

| 種類 | 役割 | 例 |
| --- | --- | --- |
| JDBC API | Java から DB を扱う共通インターフェース | `Connection`, `PreparedStatement`, `ResultSet` |
| PostgreSQL JDBC Driver | PostgreSQL 用の通信を実装するライブラリ | `org.postgresql.Driver` |

```text
DAOのJavaコード
  |
  | JDBC API
  v
PostgreSQL JDBC Driver
  |
  | PostgreSQL用の通信
  v
PostgreSQLサーバー
```

### なぜ必要か

Java のコードは `Connection` や `PreparedStatement` を使って DB 操作を書きます。しかし、それだけでは PostgreSQL 固有の通信方法は分かりません。PostgreSQL JDBC Driver が、Java の JDBC 呼び出しを PostgreSQL への通信に変換します。

### 間違えると何が起きるか

| 不足しているもの | 起きること |
| --- | --- |
| JDBC API の理解 | DAO のコードが何をしているか分かりにくい |
| PostgreSQL JDBC Driver | `org.postgresql.Driver` が見つからない、接続できない |
| JDBC URL | どの DB サーバーへ接続するか分からない |

### このプロジェクトで実際に使った設定

`pom.xml` で PostgreSQL JDBC Driver を dependency として追加しています。

```xml
<dependency>
    <groupId>org.postgresql</groupId>
    <artifactId>postgresql</artifactId>
    <version>42.7.11</version>
</dependency>
```

DAO では `Class.forName("org.postgresql.Driver")` と `DriverManager.getConnection(...)` を使っています。

## 10. Mavenとpom.xmlの役割

### 結論

Maven は、必要なライブラリの取得とビルド設定を管理する道具です。`pom.xml` は Maven に渡す設計書です。

### 仕組み

`pom.xml` には、プロジェクト名、Java バージョン、依存ライブラリ、成果物形式を書きます。

| 設定 | このプロジェクトの値 | 意味 |
| --- | --- | --- |
| `groupId` | `com.example` | プロジェクトの所属名 |
| `artifactId` | `inventory-web` | 成果物名 |
| `version` | `1.0.0` | バージョン |
| `packaging` | `war` | Web アプリとしてまとめる |
| `maven.compiler.source` | `1.8` | Java 8 の文法で書く |
| `maven.compiler.target` | `1.8` | Java 8 向けの class を作る |

### なぜ必要か

Maven がない場合、Servlet API、JSTL、PostgreSQL JDBC Driver などの jar を手作業で探して、Eclipse や Tomcat に入れる必要があります。Maven を使うと、`pom.xml` に書いた依存関係をもとに管理できます。

### 間違えると何が起きるか

| 間違い | 起きること |
| --- | --- |
| dependency が不足 | import が解決できない |
| Java バージョン指定が違う | 想定外のバージョンでコンパイルされる |
| packaging が違う | Tomcat に配置する成果物として扱いにくい |

### このプロジェクトで実際に使った設定

このプロジェクトの `pom.xml` は、`javax.servlet-api`、`jstl`、`postgresql` を依存関係として持ち、`war` を生成する設定です。

## 11. dependencyとは何か

### 結論

dependency は、このプロジェクトが自分だけでは持っていない外部ライブラリです。

### 仕組み

Maven の dependency は、主に3つの情報でライブラリを指定します。

```xml
<dependency>
    <groupId>org.postgresql</groupId>
    <artifactId>postgresql</artifactId>
    <version>42.7.11</version>
</dependency>
```

| 要素 | 意味 |
| --- | --- |
| `groupId` | ライブラリを提供するグループ |
| `artifactId` | ライブラリ名 |
| `version` | 使用するバージョン |

### なぜ必要か

Servlet、JSP の JSTL、PostgreSQL 接続は、Java 標準だけでは足りません。外部ライブラリを dependency として追加することで、コードからそれらを使えるようになります。

### 間違えると何が起きるか

```text
The import javax.servlet cannot be resolved
The import org.postgresql cannot be resolved
ClassNotFoundException: org.postgresql.Driver
```

このようなエラーは、必要な dependency がない、バージョンが合わない、Maven の依存関係が Eclipse に反映されていない場合に起きます。

### このプロジェクトで実際に使った設定

| dependency | 目的 |
| --- | --- |
| `javax.servlet-api:4.0.1` | Servlet / Filter のコードをコンパイルする |
| `jstl:1.2` | JSP で JSTL を使う |
| `postgresql:42.7.11` | PostgreSQL に JDBC 接続する |

## 12. javax.servlet-apiのscopeがprovidedである理由

### 結論

`javax.servlet-api` は `provided` にします。理由は、実行時の Servlet API は Tomcat が提供するからです。

### 仕組み

```text
コンパイル時
  |
  | Mavenの javax.servlet-api を参照
  v
Servlet / Filter のコードがコンパイルできる

実行時
  |
  | Tomcat 9 が Servlet API を提供
  v
Servlet / Filter がTomcat上で動く
```

`provided` は、「開発中とコンパイル時には必要だが、WAR には入れない」という意味です。

### なぜ必要か

Tomcat 自身が Servlet API を持っています。WAR の中にも Servlet API を入れてしまうと、Tomcat 側とアプリ側で同じ種類の API が重複します。

### 間違えると何が起きるか

| 設定ミス | 起きること |
| --- | --- |
| dependency 自体がない | `javax.servlet` の import が解決できない |
| `provided` にしない | Tomcat が持つ API と重複して、実行時トラブルの原因になる |
| Tomcat 10 と混ぜる | `javax.servlet` と `jakarta.servlet` が合わない |

### このプロジェクトで実際に使った設定

```xml
<dependency>
    <groupId>javax.servlet</groupId>
    <artifactId>javax.servlet-api</artifactId>
    <version>4.0.1</version>
    <scope>provided</scope>
</dependency>
```

## 13. packagingがwarである理由

### 結論

このプロジェクトは Tomcat に配置する Web アプリなので、`packaging` は `war` です。

### 仕組み

WAR は Web Application Archive の略で、Java Web アプリを Tomcat に渡せる形にまとめたものです。

```text
inventory-web.war
  |
  +-- WEB-INF/
  |    +-- classes/
  |    +-- lib/
  |
  +-- index.jsp
```

### なぜ必要か

Tomcat は、Servlet、JSP、依存ライブラリを一定の Web アプリ構造として読み込みます。WAR はその構造をまとめた形式です。

### 間違えると何が起きるか

| packaging | 起きること |
| --- | --- |
| `war` | Tomcat に載せる Web アプリとして扱える |
| `jar` | 通常の Java ライブラリやアプリ形式になり、Tomcat 用としては目的がずれる |

### このプロジェクトで実際に使った設定

```xml
<packaging>war</packaging>
```

最終的なファイル名は `pom.xml` の `finalName` により `inventory-web` になります。

## 14. EclipseにTomcat Runtimeを登録する意味

### 結論

Eclipse に Tomcat Runtime を登録するのは、Eclipse に「どの Tomcat でこの Web アプリを動かすか」を教えるためです。

### 仕組み

```text
Eclipse
  |
  | Tomcatの場所とバージョンを知る
  v
Tomcat Runtime
  |
  | プロジェクトを追加して起動
  v
inventory-web-study
```

### なぜ必要か

Eclipse は Java のコードを書く道具ですが、それだけでは Servlet をブラウザから動かせません。Tomcat Runtime を登録することで、Eclipse から Tomcat を起動し、Web アプリとして実行できます。

### 間違えると何が起きるか

| 状態 | 起きること |
| --- | --- |
| Runtime 未登録 | Eclipse から Tomcat にプロジェクトを追加できない |
| Tomcat 10 を登録 | `javax.servlet` プロジェクトと合わない |
| Java バージョンがずれる | 起動やビルド時の問題が分かりにくくなる |

### このプロジェクトで実際に使った設定

Tomcat 9 を Runtime として登録する前提です。Tomcat のインストール場所は環境によって異なります。

## 15. Publishすると何が起きるか

### 結論

Publish は、Eclipse のプロジェクト内容を Tomcat が実行する場所へ反映する操作です。

### 仕組み

```text
Eclipseワークスペース
  |
  | Publish
  v
Eclipseが管理するTomcat用の配置場所
  |
  v
TomcatがWebアプリとして実行
```

Publish では、Java のコンパイル結果、JSP、`WEB-INF` 配下、必要なライブラリなどが実行用の場所に反映されます。

### なぜ必要か

Eclipse 上でファイルを直しても、Tomcat が見ている実行用ファイルに反映されていなければ、ブラウザで見える動きは変わりません。

### 間違えると何が起きるか

| 状態 | 起きること |
| --- | --- |
| Publish していない | 修正したのにブラウザでは古い動きのまま |
| Java クラス変更後に再起動していない | 変更が反映されない場合がある |
| 依存関係が反映されていない | 実行時にクラスが見つからないことがある |

### このプロジェクトで実際に使った設定

Eclipse から Tomcat 9 にこの WAR 形式の Web アプリを Publish する前提です。Publish の配置先は Eclipse の設定によって異なります。

## 16. ブラウザからServletまでリクエストが届く流れ

### 結論

ブラウザのリクエストは、Tomcat、Filter、Servlet、DAO、JSP という順に処理されます。Filter は Servlet より前に動きます。

### 仕組み

商品一覧画面を例にすると、流れは次のようになります。

```text
ブラウザ
  |
  | GET /inventory-web/...
  v
Tomcat
  |
  v
Filter
  |
  v
Servlet
  |
  v
DAO
  |
  v
PostgreSQL
  |
  v
Servlet
  |
  v
JSP
  |
  v
HTML
  |
  v
ブラウザ
```

### なぜ必要か

この流れを理解すると、エラーが起きたときにどこを見るべきか分かります。

| 問題 | 見る場所 |
| --- | --- |
| 画面が開かない | Tomcat、URL、ポート |
| ログイン前に弾かれる | LoginFilter |
| POST が 403 になる | CsrfFilter、CSRF トークン |
| DB エラーになる | DAO、JDBC URL、PostgreSQL |
| 表示が崩れる | JSP、JSTL、HTML |

### このプロジェクトで実際に使った設定

`LoginFilter` と `CsrfFilter` は Servlet より前に動きます。`CsrfFilter` は POST リクエストで `csrfToken` を確認します。

## 17. よく起きた問題と原因

### PostgreSQLサービスが停止していた

#### 結論

Tomcat が起動していても、PostgreSQL が停止していると DB を使う処理は失敗します。

#### 仕組み

```text
ブラウザ
  |
  v
Tomcat 起動済み
  |
  v
DAO
  |
  v
PostgreSQL 停止中
  |
  v
DB接続エラー
```

#### なぜ必要か

Tomcat は Web アプリを動かすソフトで、PostgreSQL はデータを保存するソフトです。片方を起動しても、もう片方が自動で起動するとは限りません。

#### 間違えると何が起きるか

ログイン、商品一覧、商品登録など、DB を使う処理で接続エラーになります。エラーメッセージは環境によって異なります。

#### このプロジェクトで確認すること

PostgreSQL 18 のサービスが起動しているかを確認します。確認方法は Windows のサービス管理やインストール方法によって異なります。

### PKIX path building failed

#### 結論

`PKIX path building failed` は、Java が HTTPS 接続先の証明書を信頼できないときに起きるエラーです。

#### 仕組み

```text
Java / Maven
  |
  | HTTPSで外部へ接続
  v
接続先サーバー
  |
  v
証明書を検証できない
  |
  v
PKIX path building failed
```

#### なぜ必要か

Maven が依存ライブラリを取得するとき、HTTPS 通信を使います。Java は安全のため、接続先の証明書を検証します。

#### 間違えると何が起きるか

dependency の取得に失敗し、Eclipse や Maven で必要なライブラリがそろわないことがあります。

#### このプロジェクトでの扱い

原因や対処は環境によって異なります。JDK の証明書ストア、社内ネットワーク、プロキシ、Maven が使っている Java バージョンなどを確認します。実際に試していない証明書追加手順は、このドキュメントでは断定しません。

### Java 8とJava 25の使い分け

#### 結論

このプロジェクトでは Java 8 を使います。Java 25 が PC に入っていても、Eclipse、Maven、Tomcat がどの Java を使っているかを分けて確認します。

#### 仕組み

| 場面 | 確認すること |
| --- | --- |
| Eclipse | `JRE System Library [JavaSE-1.8]` か |
| Maven | `pom.xml` の `source` / `target` が `1.8` か |
| Tomcat | Tomcat が Java 8 前提で動くか |
| コマンドライン | `java -version` の結果 |

#### なぜ必要か

同じプロジェクトでも、Eclipse と Maven と Tomcat が別々の Java を使っていると、エラーの原因が追いにくくなります。

#### 間違えると何が起きるか

```text
Eclipseではエラーなし
  |
  v
Mavenではエラー

または

Mavenでは成功
  |
  v
Tomcat実行時にエラー
```

#### このプロジェクトで実際に使った設定

`pom.xml` は Java 8 対象です。Eclipse 側も JavaSE-1.8 として扱います。

### Tomcat 9と10の混同

#### 結論

このプロジェクトは Tomcat 9 です。Tomcat 10 にすると `javax.servlet` と `jakarta.servlet` の違いで合わなくなります。

#### 仕組み

```text
inventory-web-study
  |
  | javax.servlet
  v
Tomcat 9

Tomcat 10
  |
  | jakarta.servlet
  v
別系統
```

#### なぜ必要か

Servlet と Filter は Tomcat が呼びます。アプリ側と Tomcat 側の Servlet API が同じ系統でないと動きません。

#### 間違えると何が起きるか

ビルドエラー、クラスが見つからないエラー、デプロイ失敗などが起きる可能性があります。

#### このプロジェクトで実際に使った設定

Java コードと `pom.xml` は `javax.servlet` 系です。Tomcat 9 を使います。

### CsrfFilterがgetParameterを先に呼んだことで起きたPOSTの日本語文字化け

#### 結論

このプロジェクトでは、POST の日本語文字化けの原因は `CsrfFilter` が `getParameter("csrfToken")` を先に呼んだことでした。対策として、`CsrfFilter` 内で `getParameter()` より前に `request.setCharacterEncoding("UTF-8")` を呼ぶように修正しました。

#### 仕組み

POST のフォームデータは、最初に `getParameter()` が呼ばれたタイミングで解析されます。文字コードを指定するなら、その前に行う必要があります。

```text
正しい順序

CsrfFilter
  |
  | request.setCharacterEncoding("UTF-8")
  v
request.getParameter("csrfToken")
  |
  v
Servlet側のgetParameter()
```

問題が起きていた順序は次の考え方です。

```text
問題が起きる順序

CsrfFilter
  |
  | request.getParameter("csrfToken")
  v
リクエスト本文が先に解析される
  |
  v
Servlet
  |
  | request.setCharacterEncoding("UTF-8")
  v
すでに解析済みなので遅い
```

#### なぜ必要か

CSRF 対策では、POST のたびに `csrfToken` を確認します。そのため `CsrfFilter` は Servlet より先に `getParameter("csrfToken")` を呼びます。ここで文字コード設定をしていないと、商品名などの日本語フォーム値も誤った文字コードで解析される可能性があります。

#### 間違えると何が起きるか

商品名など、POST された日本語が文字化けします。Servlet 側で後から `request.setCharacterEncoding("UTF-8")` を呼んでも、すでにパラメータ解析済みなら間に合いません。

#### このプロジェクトで実際に使った設定

現在の `CsrfFilter` では、`getParameter("csrfToken")` より前に文字コードを設定しています。

```java
/*
 * 必ずgetParameter()より前に実行する。
 */
httpRequest.setCharacterEncoding("UTF-8");
```

この修正により、CSRF トークン確認より前に POST 本文の文字コードを指定できます。

### psqlのclient_encodingによる表示上の文字化け

#### 結論

`psql` の文字化けは、DB のデータが壊れているとは限りません。今回の Windows 端末では、`psql` で `\encoding SJIS` にすると正常表示されました。

#### 仕組み

PostgreSQL と `psql` の間では、`client_encoding` に従って文字がやり取りされます。さらに、Windows 端末側の文字コードとも表示が関係します。

```text
PostgreSQL内のデータ
  |
  | client_encoding
  v
psql
  |
  | 端末の文字コードで表示
  v
Windows端末
```

#### なぜ必要か

DB 内のデータ、PostgreSQL と `psql` の通信、端末の表示文字コードは別々に考える必要があります。画面上で文字化けしても、すぐにデータが壊れたと判断しないためです。

#### 間違えると何が起きるか

| 対応 | 注意点 |
| --- | --- |
| `\encoding UTF8` にする | ターミナル側も UTF-8 にそろっていないと表示が崩れることがある |
| `\encoding SJIS` にする | Windows 端末では正常表示される場合がある |
| 常に UTF8 にすればよいと考える | 端末側と合わず、逆に文字化けすることがある |

#### このプロジェクトで実際に確認したこと

今回の Windows 端末では、`psql` で次のようにすると日本語が正常表示されました。

```text
\encoding SJIS
```

UTF-8 を使う場合は、PostgreSQL / `psql` だけでなく、ターミナル側も UTF-8 にそろえる必要があります。環境によって正しい表示設定は異なるため、「常に UTF8 に変更すればよい」とは断定しません。

## 初期設定でそろえるべきこと

このプロジェクトでは、次の組み合わせを崩さないことが重要です。

| 項目 | このプロジェクトの値 | 理由 |
| --- | --- | --- |
| Java | Java 8 | `pom.xml` が Java 8 対象だから |
| Eclipse | JavaSE-1.8 | IDE 上でも Java 8 として扱うため |
| Tomcat | Tomcat 9 | `javax.servlet` を使うため |
| Servlet API | `javax.servlet-api:4.0.1` | Tomcat 9 と対応するため |
| Servlet API scope | `provided` | 実行時は Tomcat が提供するため |
| packaging | `war` | Tomcat に Web アプリとして配置するため |
| DB | PostgreSQL 18 | 商品やユーザー情報を保存するため |
| DB 接続 | `localhost:5432/postgres` | DAO の JDBC URL で使っているため |

設定値は、実際のコードと `pom.xml` にあるものを優先して確認します。環境によって変わる値は、推測で固定せず「環境によって異なる」と扱います。
