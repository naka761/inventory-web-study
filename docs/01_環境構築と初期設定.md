# 01. 環境構築と初期設定

この章は、`inventory-web-study`を数か月後に読み返したときに、単に「このボタンを押した」ではなく、**なぜその設定が必要だったのか、誰と誰をつないでいるのか、設定を間違えると何が起きるのか**を思い出すための記録です。

このプロジェクトでは、次の役割を持つソフトウェアを組み合わせています。

```text
ブラウザ
  ↓ HTTP
Apache Tomcat 9
  ↓ Servletを呼び出す
Java Webアプリ
  ↓ JDBC
PostgreSQL 18
```

EclipseとMavenは、この仕組みを**作成・コンパイル・配置するための開発道具**です。

## 2. 最初に登場人物を整理する

環境構築で混乱しやすい理由は、Java、Eclipse、Tomcat、Maven、PostgreSQLが同時に登場するからです。それぞれは別の仕事をしています。

| 登場人物 | このプロジェクトでの仕事 |
| --- | --- |
| ブラウザ | URLへHTTPリクエストを送り、返ってきたHTMLを表示する |
| Eclipse | JavaやJSPを編集し、エラー表示やTomcat操作を支援する |
| JDK 8 | Javaソースをコンパイルし、Javaプログラムを実行する |
| Maven | 必要なjarとビルド設定を`pom.xml`から管理する |
| Tomcat 9 | HTTPを受け取り、Filter・Servlet・JSPを実行する |
| PostgreSQL | 商品、ユーザー、操作履歴を保存・検索する |
| JDBC API | JavaからDBを操作する共通ルール |
| PostgreSQL JDBC Driver | JDBCの命令をPostgreSQL用通信へ変換する |

一言で表すと、次のようになります。

```text
Eclipse：作業机
JDK：Javaを作って動かす道具
Maven：必要な部品を集める係
Tomcat：Webから来た依頼をJavaへ渡す係
PostgreSQL：データを保存する別サーバー
JDBC Driver：Java語とPostgreSQL語の通訳
```

## 3. JDK、JRE、Javaの関係

### 3.1 結論

このプロジェクトの開発とTomcatの実行には、**Temurin JDK 8**を使用します。

### 3.2 用語の違い

| 用語 | 意味 |
| --- | --- |
| Java | 言語、実行環境、関連技術全体を指す言葉 |
| JRE | コンパイル済みJavaプログラムを実行する環境 |
| JDK | JRE相当の機能に加え、コンパイラなど開発道具を含む一式 |

Javaソースは、そのままCPUが実行するわけではありません。

```text
Product.java
  ↓ javacでコンパイル
Product.class
  ↓ JVMが実行
プログラムが動く
```

### 3.3 JDKが必要な理由

Eclipseでコードを書くには、少なくとも次の処理が必要です。

- `.java`を`.class`へコンパイルする
- Java標準クラスを参照する
- Tomcat自身をJavaプログラムとして実行する

JDKが正しく設定されていないと、次のような問題が起きます。

```text
StringやListまでエラーになる
Javaをコンパイルできない
Tomcatを起動できない
想定外のJavaバージョンで動く
```

## 4. Java 25があるのにJava 8を使う理由

### 4.1 結論

PC全体の既定Javaは25でも、このプロジェクトでは**EclipseのプロジェクトとTomcatをJava 8へ固定**します。

### 4.2 新しいJavaが悪いわけではない

Java 25の方が新しいため、一般には新機能や改善があります。しかし、このプロジェクトの目的は、会社研修で想定されるJava 8環境を再現することです。

環境が違うと、次の問題が起きます。

```text
Java 25でしか使えない文法・APIを書く
  ↓
Java 8の会社環境で動かない
```

または、

```text
Eclipse：Java 8
Maven：Java 25
Tomcat：Java 25
  ↓
場所によって結果が違い、原因が分かりにくい
```

### 4.3 `pom.xml`のJava 8設定

このプロジェクトでは、次の設定があります。

```xml
<maven.compiler.source>1.8</maven.compiler.source>
<maven.compiler.target>1.8</maven.compiler.target>
```

- `source`：Java 8の文法としてコンパイルする
- `target`：Java 8で実行できるclass形式を生成する

ただし、これだけですべてがJava 8になるわけではありません。**Eclipseが使うJDK、Mavenを動かすJDK、Tomcatを動かすJDKも確認する必要があります。**

### 4.4 実際に確認したコマンド

Temurin JDK 8を直接指定して確認しました。

```powershell
& "C:\Program Files\Eclipse Adoptium\jdk-8.0.492.9-hotspot\bin\java.exe" -version
& "C:\Program Files\Eclipse Adoptium\jdk-8.0.492.9-hotspot\bin\javac.exe" -version
```

確認できた内容は、概ね次の通りです。

```text
openjdk version "1.8.0_492"
javac 1.8.0_492
```

## 5. Eclipseの`JRE System Library [JavaSE-1.8]`

### 5.1 結論

`JRE System Library [JavaSE-1.8]`は、Eclipseに対して、

> このプロジェクトはJava 8の標準ライブラリとルールで扱う

と指定する設定です。

### 5.2 何が入っているか

たとえば次のクラスは、外部jarではなくJava標準ライブラリです。

```java
java.lang.String
java.util.List
java.sql.Connection
```

Eclipseは`JRE System Library`を参照して、これらのクラスや利用可能なAPIを判断します。

### 5.3 間違えると何が起きるか

| 状態 | 起きること |
| --- | --- |
| JavaSE-1.8 | Java 8前提で編集・検査する |
| JavaSE-25 | Java 8では使えない機能を書ける可能性がある |
| 壊れている／未設定 | 標準クラスまで解決できないことがある |

これはTomcatの設定とは別です。

```text
JRE System Library：Eclipseがこのプロジェクトをどう理解するか
Tomcat Runtime：EclipseがどのTomcatでWebアプリを動かすか
```

## 6. Tomcatとは何か

### 6.1 結論

Tomcatは、ブラウザから届いたHTTPリクエストを受け取り、該当するFilter・Servlet・JSPを実行するソフトです。

普通のJavaクラスへ、ブラウザが直接アクセスすることはできません。

```text
ブラウザ ──直接──> ProductListServlet
```

ではなく、実際には次の流れです。

```text
ブラウザ
  ↓ HTTP
Tomcat
  ↓ URLに対応するクラスを探す
Filter
  ↓
Servlet
```

### 6.2 Tomcatの主な仕事

- ポート8080でHTTPを待ち受ける
- URLとServletを対応付ける
- `HttpServletRequest`を作る
- `HttpServletResponse`を作る
- Filterを実行する
- Servletの`doGet()`や`doPost()`を呼ぶ
- JSPからHTMLを生成する
- `HttpSession`を管理する
- WAR形式のWebアプリを配備する

### 6.3 Tomcatが止まっていると

ブラウザから次を開いても接続できません。

```text
http://localhost:8080/inventory-web/
```

PostgreSQLが正常でも、Tomcatが止まっていればWeb画面には到達できません。

## 7. Tomcat 9とTomcat 10を混ぜてはいけない理由

### 7.1 結論

このプロジェクトは`javax.servlet`を使っているため、**Tomcat 9**で動かします。

### 7.2 パッケージ名の違い

| 世代 | 主なパッケージ |
| --- | --- |
| Tomcat 9 | `javax.servlet.*` |
| Tomcat 10以降 | `jakarta.servlet.*` |

例：

```java
// Tomcat 9系
import javax.servlet.http.HttpServlet;

// Tomcat 10系
import jakarta.servlet.http.HttpServlet;
```

Javaではパッケージ名まで含めてクラスの正体が決まるため、両者は別物です。

### 7.3 このプロジェクトの証拠

Javaコードでは`javax.servlet`をimportしています。また、`pom.xml`も次の依存関係です。

```xml
<dependency>
    <groupId>javax.servlet</groupId>
    <artifactId>javax.servlet-api</artifactId>
    <version>4.0.1</version>
    <scope>provided</scope>
</dependency>
```

したがって、組み合わせは次です。

```text
Java 8
+
javax.servlet-api 4.0.1
+
Tomcat 9
```

### 7.4 実際に起きた混乱

PCにはTomcat 10.1系も存在しました。しかし、それは別の実験用です。

このプロジェクトでTomcat 10を選ぶと、次のような問題が起き得ます。

- `javax.servlet`のクラスを実行環境が提供しない
- Servletとして認識されない
- importを`jakarta`へ全部変更する必要が出る
- どちらのTomcatへPublishしたのか分からなくなる

## 8. `localhost`、8080、5432の違い

### 8.1 `localhost`

`localhost`は、自分のPC自身を表す名前です。

```text
localhost ＝ 今操作しているこのPC
```

### 8.2 ポート番号

同じPCでは複数のサーバーが動くため、ポート番号で窓口を分けます。

| 接続先 | ポート | 用途 |
| --- | ---: | --- |
| Tomcat | 8080 | ブラウザからWebアプリへ接続 |
| PostgreSQL | 5432 | JavaやpsqlからDBへ接続 |

```text
localhost:8080 → Tomcatの窓口
localhost:5432 → PostgreSQLの窓口
```

### 8.3 URLの比較

ブラウザからWebアプリへ：

```text
http://localhost:8080/inventory-web/products
```

JavaからPostgreSQLへ：

```text
jdbc:postgresql://localhost:5432/postgres
```

`localhost`は同じでも、通信方式とポートが異なるため別のソフトにつながります。

## 9. PostgreSQLはJavaとは別のサーバー

### 9.1 結論

PostgreSQLは、EclipseやTomcatの中に存在するわけではありません。Windows上で別サービスとして動いています。

```text
Tomcat上のJavaアプリ
  ↓ ネットワーク接続
PostgreSQLサービス
```

### 9.2 このPCでのサービス名

```text
postgresql-x64-18
```

管理者PowerShellで起動する例：

```powershell
Start-Service postgresql-x64-18
```

状態確認：

```powershell
Get-Service postgresql-x64-18
```

ポート確認：

```powershell
Test-NetConnection localhost -Port 5432
```

### 9.3 停止していると何が起きるか

```text
Tomcat：起動中
PostgreSQL：停止中
```

この場合、ログイン画面自体は表示できることがあります。しかし、UserDAOがDBへ接続する段階で失敗します。

```text
ブラウザ → Tomcat：成功
Tomcat → PostgreSQL：失敗
```

「画面が開いたから全部起動している」とは限りません。

## 10. psqlとは何か

`psql`はPostgreSQLへSQLを送るためのコマンドラインクライアントです。

このPCでは次の場所にあります。

```text
C:\Program Files\PostgreSQL\18\bin\psql.exe
```

接続例：

```powershell
& "C:\Program Files\PostgreSQL\18\bin\psql.exe" `
  -h localhost `
  -p 5432 `
  -U postgres `
  -d postgres
```

各指定の意味：

| 指定 | 意味 |
| --- | --- |
| `-h localhost` | 接続先PC |
| `-p 5432` | PostgreSQLのポート |
| `-U postgres` | DBユーザー |
| `-d postgres` | 接続するデータベース |

`psql`はDBサーバー本体ではありません。

```text
psql：PostgreSQLへ命令する操作端末
PostgreSQL：命令を受けてデータを処理するサーバー
```

## 11. JDBC APIとPostgreSQL JDBC Driver

### 11.1 結論

JDBC APIはJava共通の操作ルールで、PostgreSQL JDBC DriverはそのルールをPostgreSQL用に実行する部品です。

### 11.2 JDBC API

Javaコードでは次の型を利用しています。

```java
Connection
PreparedStatement
ResultSet
DriverManager
```

これらは「接続する」「SQLを準備する」「結果を読む」という共通ルールです。

### 11.3 PostgreSQL JDBC Driver

実際にPostgreSQLと通信する方法は、PostgreSQL JDBC Driverが担当します。

```text
ProductDAO
  ↓ JDBC API
PostgreSQL JDBC Driver
  ↓ PostgreSQL用の通信
PostgreSQL
```

ドライバがなければ、JavaはPostgreSQL固有の通信方法を知りません。

### 11.4 このプロジェクトの依存関係

```xml
<dependency>
    <groupId>org.postgresql</groupId>
    <artifactId>postgresql</artifactId>
    <version>42.7.11</version>
</dependency>
```

コード内では念のため次も実行しています。

```java
Class.forName("org.postgresql.Driver");
```

現代のJDBCでは自動検出される場合がありますが、このプロジェクトでは「どのドライバを使っているか」を明示的に理解するため残しています。

## 12. Mavenとは何か

### 12.1 結論

Mavenは、必要なjarを手作業で集める代わりに、`pom.xml`の内容から依存関係とビルド方法を管理するツールです。

### 12.2 Mavenを使わない場合

手作業では次のような作業が必要です。

```text
Servlet APIのjarを探す
JSTLのjarを探す
PostgreSQL Driverのjarを探す
正しいバージョンを選ぶ
EclipseのBuild Pathへ追加する
WARへ入れるjarと入れないjarを判断する
```

Mavenでは、必要な部品を`pom.xml`へ記載します。

### 12.3 `pom.xml`が管理するもの

このプロジェクトでは、主に次を管理します。

- プロジェクト名
- バージョン
- Java 8設定
- WAR形式
- Servlet API
- JSTL
- PostgreSQL JDBC Driver
- WAR作成プラグイン

### 12.4 EclipseのMavenと`mvn`コマンドは別に考える

EclipseにはMaven連携機能があり、Eclipse上では依存関係が解決できることがあります。一方、PowerShellで次を実行するには、コマンドライン用MavenがPATHから見つかる必要があります。

```powershell
mvn package
```

Codexによる確認時には、`mvn`コマンドが見つからない環境がありました。これは即座に「プロジェクトが壊れている」という意味ではありません。

```text
EclipseのMaven連携：利用可能
コマンドラインのmvn：未導入またはPATH未設定
```

という状態はあり得ます。

## 13. `dependency`とは何か

`dependency`は、このプロジェクトが必要とする外部ライブラリです。

このプロジェクトには、主に3つあります。

| dependency | 必要な理由 |
| --- | --- |
| `javax.servlet-api` | Servlet、Filter、Sessionなどをコンパイルする |
| `jstl` | JSPで`c:forEach`や`c:out`を使う |
| `postgresql` | JavaからPostgreSQLへ接続する |

例：

```xml
<dependency>
    <groupId>javax.servlet</groupId>
    <artifactId>jstl</artifactId>
    <version>1.2</version>
</dependency>
```

| 要素 | 意味 |
| --- | --- |
| `groupId` | 提供元・グループの識別名 |
| `artifactId` | ライブラリ名 |
| `version` | 使用する版 |

依存関係が解決できないと、たとえば次のエラーが出ます。

```text
The import javax.servlet cannot be resolved
The import org.postgresql cannot be resolved
ClassNotFoundException: org.postgresql.Driver
```

## 14. `scope`が`provided`である理由

Servlet APIには次の指定があります。

```xml
<scope>provided</scope>
```

意味は次です。

```text
コンパイル時：必要なので使う
実行時：Tomcatが提供するのでWARへ入れない
```

Tomcat自身がServlet APIを持っています。アプリ側にも同じものを詰めると、Tomcat側とアプリ側で同じ役割のクラスが重複し、競合の原因になります。

```text
Servlet API → Tomcatから提供される → provided
JSTL       → アプリが使う          → WARへ含める
JDBC Driver→ アプリが使う          → WARへ含める
```

## 15. なぜ`packaging`が`war`なのか

### 15.1 結論

このプロジェクトはTomcatへ配置するWebアプリなので、成果物をWAR形式にします。

```xml
<packaging>war</packaging>
```

### 15.2 JARとの違い

| 形式 | 主な用途 |
| --- | --- |
| JAR | Javaクラスや通常のJavaアプリをまとめる |
| WAR | Servlet、JSP、`WEB-INF`などWebアプリ一式をまとめる |

WARのイメージ：

```text
inventory-web.war
├─ index.jsp
└─ WEB-INF
   ├─ classes
   ├─ lib
   └─ jsp
```

Spring Bootの自己実行型アプリではJARを使う構成もありますが、今回の構成は外部Tomcatへ載せるためWARです。

## 16. `finalName`とコンテキストパス

`pom.xml`には次があります。

```xml
<finalName>inventory-web</finalName>
```

これにより、WAR名は概ね次になります。

```text
inventory-web.war
```

Tomcatへ配置されたとき、通常はWAR名がコンテキストパスの元になります。

```text
http://localhost:8080/inventory-web/
```

URLを分解すると次の通りです。

```text
http://            通信方式
localhost          接続先PC
:8080              Tomcatのポート
/inventory-web     Webアプリの区画
/products          ServletなどのURL
```

Eclipseのサーバー設定やプロジェクト設定によってコンテキストパスは変更できるため、常にWAR名と一致するとは限りません。

## 17. `failOnMissingWebXml`が`false`である理由

`pom.xml`には次があります。

```xml
<failOnMissingWebXml>false</failOnMissingWebXml>
```

昔のServletでは、URLとServletの対応などを`WEB-INF/web.xml`へ書く方法が中心でした。

このプロジェクトでは、次のようにアノテーションを使っています。

```java
@WebServlet("/products")
@WebFilter("/*")
```

そのため、`web.xml`がなくても動かせます。

この設定は、

> `web.xml`がないことだけを理由にWAR作成を失敗させない

という意味です。

EclipseへTomcat Runtimeを登録するとは、

をEclipseへ教えることです。

```text

Eclipseからサーバーを起動できない
```

## 19. EclipseのPublishとは何か

### 19.1 結論

Publishは、Eclipse上のプロジェクト変更を、Eclipseが管理するTomcat実行領域へ反映する操作です。

```text
Eclipseのプロジェクト
  ↓ コンパイル・コピー
Eclipse管理のTomcat配置領域
  ↓
Tomcatが実行
```

### 19.2 なぜ保存だけでは反映されないことがあるのか

Eclipseで編集している場所と、Tomcatが実際に読み込む場所が別だからです。

```text
編集場所：eclipse-workspace\inventory-web
実行場所：EclipseがServers用に管理する領域
```

そのため、

```text
ファイルを保存した
しかしブラウザでは古い画面のまま
```

という場合、PublishやTomcat再起動が必要なことがあります。

### 19.3 再起動が必要になる場合

JSPだけなら比較的すぐ反映されることがあります。Javaクラス、Filter、アノテーション、依存関係などの変更では、再起動が必要になる場合があります。

厳密な挙動はEclipseの設定やサーバー状態により異なります。

## 20. ブラウザからServletへ届くまで

商品一覧の例：

```text
1. ChromeでURLを開く
   GET /inventory-web/products
          ↓
2. localhost:8080のTomcatが受信
          ↓
3. このURLがinventory-web宛てだと判断
          ↓
4. Filterを実行
          ↓
5. @WebServlet("/products")のServletを探す
          ↓
6. ProductListServlet.doGet()を呼ぶ
          ↓
7. ProductDAOがPostgreSQLへSELECT
          ↓
8. ServletがProductのListをrequestへ保存
          ↓
9. JSPへforward
          ↓
10. JSPがHTMLを生成
          ↓
11. TomcatがChromeへ返す
```

環境構築の役割だけを見ると、次のつながりです。

```text
ブラウザ
  ↓ 8080
Tomcat 9（JDK 8で実行）
  ↓ コンパイル済みJava
Servlet / DAO
  ↓ JDBC Driver
PostgreSQL 18（5432）
```

## 21. 文字コード設定と、実際に起きた文字化け

文字化けは、すべて同じ原因ではありません。このプロジェクトでは、別々の2種類を経験しました。

### 21.1 Webフォーム登録時にDBへ文字化けして保存された問題

#### 症状

ブラウザで次を登録しました。

```text
トランザクション確認
```

しかし商品一覧では、次のような文字になりました。

```text
ãã©ã³...
```

これは表示だけではなく、リクエストを読む段階で文字化けした値がDBへ保存された状態でした。

#### 原因

`CsrfFilter`はPOST時に次を呼びます。

```java
httpRequest.getParameter("csrfToken");
```

`getParameter()`を最初に呼んだ時点で、Tomcatはフォーム本文全体を解析します。

Servlet側にも次はありました。

```java
request.setCharacterEncoding("UTF-8");
```

しかし、Filterの`getParameter()`より後なので手遅れでした。

```text
CsrfFilter.getParameter()
  ↓ フォーム全体を誤った文字コードで解析
Servlet.setCharacterEncoding("UTF-8")
  ↓ すでに解析済みなので直らない
```

#### このプロジェクトで行った修正

`CsrfFilter`内で、**最初の`getParameter()`より前**に指定しました。

```java
httpRequest.setCharacterEncoding("UTF-8");
```

現在の順序：

```java
httpRequest.setCharacterEncoding("UTF-8");

// この後で初めてgetParameterを呼ぶ
String requestToken =
        httpRequest.getParameter("csrfToken");
```

重要なのは「Servletに書いてあるか」だけではなく、**アプリ全体で最初に`getParameter()`を呼ぶ場所より前か**という点です。

### 21.2 psqlの画面だけが文字化けした問題

#### 症状

ブラウザでは日本語が正常なのに、psqlで既存の商品まで次のように見えました。

```text
ぶどうジュース
  ↓
縺ｶ縺ｩ縺・ず繝･繝ｼ繧ｹ
```

以前から正常だった商品も同時に化けたため、DB内の全データが突然壊れたとは考えにくい状態でした。

#### 原因

PostgreSQLからpsqlへ送る文字コードと、Windows端末が表示する文字コードの組み合わせが合っていませんでした。

```text
DB内のデータ
  ↓ client_encodingで変換
psql
  ↓ Windows端末が表示
画面
```

#### このPCで実際に成功した設定

psql内で次を実行しました。

```psql
\encoding SJIS
```

その後、日本語が正常表示されました。

確認：

```sql
SHOW client_encoding;
```

#### 注意

「常にUTF8へ設定すればよい」という意味ではありません。

- DB側の文字コード
- psqlの`client_encoding`
- Windows Terminal、PowerShell、コマンドプロンプト側の表示文字コード

をそろえる必要があります。

UTF-8で統一する場合は、端末側もUTF-8として正しく表示できる状態にする必要があります。このPCのその時点のpsql表示では、`SJIS`へ合わせることで正常になりました。

## 22. `PKIX path building failed`とは何だったか

### 22.1 結論

これはMavenそのものの文法エラーではなく、JavaがHTTPS接続先の証明書を信頼できなかったときに出るエラーです。

```text
Java / Maven
  ↓ HTTPSでjarを取得しようとする
外部リポジトリ
  ↓ 証明書を検証
信頼経路を作れない
  ↓
PKIX path building failed
```

### 22.2 証明書の役割

HTTPSでは、接続先が本物か、通信を暗号化できる相手かを証明書で確認します。

JavaはJDK内のtruststoreにある信頼情報を使います。

### 22.3 考えられる原因

- 社内ネットワークのプロキシが通信の間に入っている
- 会社独自のルート証明書がJavaへ登録されていない
- Eclipseが使うJavaとMavenが使うJavaが違う
- それぞれのJDKでtruststoreの内容が異なる
- 接続先証明書の経路をJavaが検証できない

単純に「証明書が古い」とは限りません。

### 22.4 重要な考え方

```text
誰がHTTPS通信しているか
  ↓
そのプログラムはどのJDKで動いているか
  ↓
そのJDKはどのtruststoreを見ているか
```

を確認する必要があります。

会社PCでは勝手に証明書を追加せず、研修担当者や管理者が指定する手順に従うべきです。
